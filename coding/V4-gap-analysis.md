# V4 方案 vs 原始诉求 — 差距分析与补充建议

> 逐条对照原始诉求（核心诉求 / 非核心诉求 / 隐含约束 / 最终目标）与 V4 最终方案，识别遗漏、过度设计和需要补充的内容。

---

## 一、原始诉求一句话

> 我想要一套尽量开箱即用的 AI 能力体系，在接收到售前或客户的模糊需求后，先自动完成需求澄清，再继续推进方案、开发、测试和交付；整个过程中只把极少数关键决策点交给人，其余尽量自主完成，并最终输出可用成果。

---

## 二、逐条对照

### P0-1：能处理模糊需求输入

**原始诉求**：输入不是标准工单，而是售前/客户给出的不完整、有歧义、信息缺失的描述。AI 需要先把模糊内容转成可执行需求。

**V4 覆盖情况**：✅ 已覆盖

- CLARIFY 阶段：Superpowers brainstorming（发散式探索问题空间）→ Flow-Next interview（收敛式精炼）
- 从模糊输入到结构化 epic spec 的完整链路

**但有一个 gap**：⚠️ **"模糊需求如何进入系统"这一步缺少具体设计**

V4 假设用户已经在 Claude Code 环境中，可以直接输入模糊需求。但原始诉求里提到的场景是"售前或客户给出的模糊需求"——这意味着需求可能以各种形态到达：

- 一段微信/Slack 消息
- 一封邮件
- 一次会议纪要
- 一份粗糙的需求文档

**补充建议**：增加一个**需求入口标准化**环节——在 CLARIFY 之前，定义一个最简输入模板或 prompt，帮助使用者把各种形态的模糊需求统一为 CLARIFY 的输入格式。不需要复杂的渠道对接（原始诉求也说了暂不重点关注入口渠道），只需要一个**"把任意文本转成 CLARIFY 输入"的标准 prompt**。

---

### P0-2：能自动做需求澄清

**原始诉求**：
- 识别不明确点
- 自动拆分：必须确认 / 可假设推进 / 可延后确认
- 尽量减少需要人工确认的数量
- 把必须问人的问题压缩到极少数关键决策点

**V4 覆盖情况**：⚠️ 部分覆盖

**已覆盖**：
- Superpowers brainstorming 可以识别问题空间
- Flow-Next interview 可以做深度 Q&A 精炼
- Decision Bundle 可以收束必须人工拍板的问题

**关键 gap**：❌ **缺少"问题分类"能力——哪些必须问、哪些可以先假设推进**

这是原始诉求中非常核心的一条隐含约束：

> AI 需要能区分"必须确认的关键点"、"可基于行业常识先假设推进的点"、"可后置确认不阻塞主流程的点"。

V4 的 Decision Bundle 只解决了"收束必须人工拍板的问题"，但**没有设计"AI 主动判断哪些问题可以不问人"的机制**。

当前 V4 的模式偏向于：AI 遇到不确定就打包到 Decision Bundle 问人。但原始诉求要的是：**AI 应该尽量自己判断，只把真正无法自主决策的问题交给人。**

**补充建议**：在 CLARIFY 阶段增加一个**决策分类机制**：

| 分类 | 处理方式 | 示例 |
|------|---------|------|
| **必须确认** | 纳入 Decision Bundle，阻断直到人回复 | 目标用户是内部还是外部？技术选型是 A 还是 B？ |
| **可假设推进** | AI 基于上下文/行业惯例做出假设，记录假设依据，不阻断 | 数据库默认用 PostgreSQL（除非特殊要求） |
| **可延后确认** | 不阻塞当前阶段，但标记为后续需确认，在 VERIFY 阶段复核 | 日志格式的具体 field 定义 |

在 Decision Bundle 中增加 `classification` 字段：

```json
{
  "id": "D-CLARIFY-01",
  "question": "数据库选型",
  "classification": "assumable",
  "assumption": "默认 PostgreSQL",
  "assumption_basis": "团队现有技术栈",
  "override_needed_by": "PLAN"
}
```

这样 Decision Bundle 不仅是"问人的问题包"，还是"AI 假设的记录包"，人在审核时可以快速确认或推翻假设。

### P0-3：人工介入点极少

**原始诉求**：
- 大部分过程 AI 自主完成
- 只有少量高价值、必须拍板的点交给人
- 人的角色是审批者/决策者/兜底者，不是全过程手工驱动者

**V4 覆盖情况**：⚠️ 有风险过度设计

**已覆盖**：
- Decision Bundle 明确收束人工决策点
- 动态强度控制可按风险等级降低流程重量
- Ralph 自治模式支持 overnight 无人运行

**风险点**：❌ **V4 的"完整路径"人工介入点可能过多**

逐条数 V4 完整路径中的人工介入点：

| 介入点 | 阶段 | 是否必需 |
|-------|------|---------|
| Decision Bundle 拍板 #1 | CLARIFY → SPEC | 中高风险必需 |
| 轻议会审查后人工确认 | SPEC | ⚠️ 议会应该自动放行，只在 HOLD/REVISE 时才需人工 |
| Decision Bundle 拍板 #2 | SPEC → PLAN | ⚠️ 如果 CLARIFY 已充分澄清，此处可省 |
| 计划议会审查后人工确认 | PLAN | ⚠️ 同上 |
| Decision Bundle 拍板 #3 | PLAN → EXECUTE | 高风险必需 |
| EXECUTE 过程中人工检查点 | EXECUTE | 高风险才需要 |
| 验收议会审查后人工确认 | VERIFY | ⚠️ 议会 PASS 应自动放行 |
| Decision Bundle 拍板 #4 | VERIFY → DONE | 必需 |
| 发布议会审查后人工确认 | DONE | 必需 |

**完整路径最多 9 个人工介入点**——这与"极少数关键决策点交给人"的诉求存在冲突。

**补充建议**：明确区分**自动放行**和**人工确认**：

| 场景 | 处理方式 |
|------|---------|
| 议会 verdict = GO/PASS/READY/RELEASE_READY | **自动放行**，不等人确认 |
| 议会 verdict = REVISE/HOLD/BLOCK/NOT_READY | 阻断，等人确认或提供补充 |
| Decision Bundle 中所有项 classification = assumable | **自动放行**（假设推进），只记录 |
| Decision Bundle 中有 classification = must_confirm | 阻断，等人拍板 |

按此设计，**低风险 feature 的人工介入点可压缩到 1~2 个**（PLAN→EXECUTE 的 Decision Bundle + DONE 的发布确认），更契合原始诉求。

### P0-4：能一路推进到交付物

**原始诉求**：
- 最终要有可执行方案、代码/系统、测试结果、可交付介质
- 目标不是"分析得好"，而是真正交付

**V4 覆盖情况**：✅ 已覆盖

- Flow-Next 的 `/flow-next:work` 可以产出代码 + 测试
- Evidence 记录 commits / tests / PRs
- DONE 阶段有交付包（release-notes + delivery-summary）

**但有一个 gap**：⚠️ **"可交付介质"的定义不够具体**

原始诉求提到的交付物包括：可执行方案、原型/流程、代码/系统、测试结果、可交付介质。V4 的 DONE 阶段产出 `release-notes.md` 和 `delivery-summary.md`，但这两个是**文档**，不是**介质**。

真正的"可交付介质"可能包括：
- 可部署的代码包 / Docker 镜像
- API 文档
- 用户操作手册
- 测试报告（含覆盖率）
- 部署指南

**补充建议**：在 DONE 阶段的产物清单中，增加**交付介质 checklist**，根据项目类型动态决定需要产出什么。这不需要新增机制，只需要在 `delivery-summary.md` 的模板中增加一个 checklist section。

### P1-1：尽量直接可用，少二次开发

**原始诉求**：希望工具尽量开箱即用。

**V4 覆盖情况**：⚠️ 需要关注

V4 的方案拆成两部分来看：

**Flow-Next 部分**：✅ 高度开箱即用
- 安装一个 Claude Code 插件即可
- `/flow-next:plan`、`/flow-next:work` 直接可用
- flowctl CLI 零配置

**stage-harness 部分**：❌ 需要大量开发
- Decision Bundle 机制需要自建
- 分层议会需要自建（reviewer agent prompt、裁决汇总逻辑）
- 阶段门禁 hooks 需要自建
- 动态强度控制需要自建
- `.harness/` 目录结构和模板需要自建

**补充建议**：**MVP-1 应该做到"Flow-Next 即可跑通全流程"**，stage-harness 的治理能力作为渐进增强。具体来说：

- MVP-0（即刻可用）：只装 Flow-Next + Superpowers，用 `/flow-next:plan` → `/flow-next:work` → `/flow-next:impl-review` 跑通全流程，**零 stage-harness 代码**
- MVP-1：加入 Decision Bundle（最核心的人工决策收束）
- MVP-2：加入议会 + ECC
- MVP-3：加入 Ralph + 动态强度

这样用户可以**从 Day 1 就开始使用**，而不是等 stage-harness 开发完才能体验价值。

### P1-2：最好是 agent / skills / AI coding 方向

**V4 覆盖情况**：✅ 完全契合

V4 全部基于 Claude Code 原生能力体系（plugins、skills、agents、hooks、subagents、agent teams）。Flow-Next 本身就是一个 Claude Code 插件。

### P1-3：能把多个阶段串起来形成闭环

**V4 覆盖情况**：✅ 已覆盖

CLARIFY → SPEC → PLAN → EXECUTE → VERIFY → FIX → DONE 的完整闭环已设计清楚。Flow-Next 的 plan → work → review 是天然的闭环。Ralph 自治循环进一步强化了闭环能力。

**V4 覆盖情况**：❌ 未充分覆盖

这条在上面 P0-2 已分析。V4 的 Decision Bundle 偏向于"把不确定的都问人"，缺少"AI 主动判断哪些可以先假设推进"的分类机制。

**已在 P0-2 中给出补充建议。**

### 隐含约束-2：不能只给建议，必须能执行

**V4 覆盖情况**：✅ 已覆盖

Flow-Next 的 `/flow-next:work` 是真正的代码执行——worker subagent 写代码、写测试、做原子提交。不是只输出方案文档。

### 隐含约束-3：需要多阶段协作能力

**V4 覆盖情况**：✅ 已覆盖

V4 有 20 个专用 agent、多阶段状态机、分层议会。多角色协作能力充足。

### 隐含约束-4：结果需要尽量稳定、可复用

**原始诉求**：希望形成固定方法、可复用流程、可复用技能，最终交给其他人也能用。

**V4 覆盖情况**：⚠️ 部分覆盖

**已覆盖**：
- Flow-Next memory 可以沉淀 pitfalls / conventions / decisions
- DONE 阶段的 learning-candidates.md 可以提名经验候选
- epic spec 模板可以作为可复用的规格格式

**gap**：❌ **缺少"能力沉淀与复制"的显式机制**

V4 设计了"学习治理"（经验提名 → 人工晋升），但没有明确回答：

1. **沉淀到哪里**？是写入 CLAUDE.md 的 rules？是 Flow-Next 的 memory？是独立的 skills 库？
2. **如何复制给其他人**？新成员如何获得之前积累的经验？
3. **如何避免规则膨胀**？经验越积越多，如何清理过时的规则？

**补充建议**：

定义一个**三层沉淀模型**：

| 层 | 存储位置 | 内容 | 生命周期 |
|----|---------|------|---------|
| **项目级** | Flow-Next `.flow/memory/` | 当前项目的 pitfalls / conventions | 项目结束时归档 |
| **团队级** | CLAUDE.md / rules | 团队通用的工程约定和经验 | 长期维护，定期清理 |
| **个人级** | 本地配置 | 个人偏好和习惯 | 个人维护 |

晋升路径：`项目经验 → 人工评审 → 团队规则`

清理策略：团队级规则不超过 30 条，每季度 review 一次。

## 三、发现的过度设计风险

对照原始诉求，V4 有几个地方存在**过度设计**的风险：

### 3.1 ~~四类议会可能过重~~ → 已修正：议会是核心机制，不做降级

完整多 reviewer 议会是核心质量保障机制，**不降级为 checklist 式**。通过动态强度控制调整议会规模（低风险 3 reviewer，高风险 7 reviewer），而不是取消议会本身。议会在 MVP-1 即实现。

### 3.2 ECC 精选子集可能过早引入

原始诉求没有提到安全审计、治理合规。ECC 的 security-reviewer 等能力虽然有价值，但对于"从模糊需求到交付"的核心链路来说不是 Day 1 必需品。

**建议**：ECC 从 MVP-2 甚至 MVP-3 再引入，MVP-1 完全不需要。

### 3.3 Eval 数据面（token / latency / outcome）可能过早引入

原始诉求关注的是"能不能跑通"，不是"跑得多高效"。量化指标收集对 MVP 来说是锦上添花。

**建议**：Eval 数据面放到 MVP-3 之后。

## 四、发现的关键遗漏

### 4.1 ❌ 缺少"端到端操作手册"

V4 是一份**架构方案**，但不是一份**操作指南**。一个新用户拿到 V4，不知道：

- 第一步怎么做？（安装什么？配置什么？）
- 拿到一个模糊需求后，具体敲什么命令？
- 每个阶段出来的东西长什么样？
- 出了问题怎么回退？

**补充建议**：编写一份 **Quick Start Guide**——用一个具体例子（比如"实现一个用户注册功能"），走一遍从模糊需求到交付的全流程，标注每一步敲什么命令、产出什么文件、人需要做什么。

### 4.2 ❌ 缺少"失败路径"设计

V4 的全部设计都围绕"成功路径"——需求澄清了、计划通过了、代码写好了、review 过了。但实际场景中：

- CLARIFY 后发现需求根本不可行怎么办？
- PLAN 后发现技术方案不成立怎么办？
- EXECUTE 中途发现依赖的外部服务不可用怎么办？
- 整个 epic 做到一半被取消了怎么办？

**补充建议**：增加一组**终止/挂起/降级**策略：

| 场景 | 处理 |
|------|------|
| 需求不可行 | CLARIFY 阶段输出 `not-feasible` 结论，记录原因，流程终止 |
| 技术方案不成立 | PLAN 阶段计划议会 verdict = BLOCK，回流 SPEC 或终止 |
| 外部依赖不可用 | task 状态标记为 `blocked`（flowctl 已支持），等待解除 |
| Epic 被取消 | flowctl 标记 epic 为 `cancelled`，保留产物作为记录 |
| Token 预算耗尽 | 暂停执行，保存 handoff 状态，等待人工续费后恢复 |

### 4.3 ❌ 缺少"多 feature 并行"的考虑

原始诉求没有明确说"一次只做一个需求"。实际场景中，可能同时收到多个模糊需求，需要并行推进。

V4 的设计是单 epic 视角——每个 epic 独立走 CLARIFY → DONE。但如果多个 epic 同时推进：

- 人工决策点可能堆积（多个 Decision Bundle 同时等待拍板）
- 议会资源可能不够（同时运行多组 reviewer agent）
- 代码仓库可能冲突（多个 epic 修改同一文件）

**补充建议**：明确声明"V4 当前按单 epic 设计，多 epic 并行通过 Flow-Next 的 worktree 隔离和 epic 间依赖（`depends_on_epics`）实现基础并行；复杂的多 epic 调度放在后续扩展"。

### 4.4 ⚠️ "开箱即用"的程度需要更明确

原始诉求强调"尽量开箱即用"。V4 需要明确告诉用户：

| 阶段 | 开箱可用度 | 需要用户做什么 |
|------|----------|-------------|
| MVP-0（Day 1） | ✅ 高 | 安装 Flow-Next + Superpowers 两个插件 |
| MVP-1 | ⚠️ 中 | 需要配置 Decision Bundle hooks（shell 脚本） |
| MVP-2 | ⚠️ 中 | 需要编写 reviewer agent prompt |
| MVP-3 | ⚠️ 低 | 需要配置 Ralph + 动态强度控制 |

**补充建议**：在 V4 文档中增加一个"开箱即用度"矩阵，让用户知道自己目前需要投入多少配置工作。

## 五、修订后的 MVP 路线（更贴合原始诉求）

原始 V4 的 MVP 路线偏向"从架构完整性出发"。对照原始诉求后，建议调整为"从用户价值出发"：

### MVP-0：Day 1 即可体验（零开发）

**目标**：用户拿到模糊需求后，当天就能跑通全流程。

**做法**：
1. 安装 Flow-Next 和 Superpowers
2. 运行 `/flow-next:setup`
3. 用 Superpowers brainstorming 澄清需求
4. 用 `/flow-next:interview` 精炼
5. 用 `flowctl epic create` + `/flow-next:plan` 生成计划
6. 用 `/flow-next:work` 执行开发
7. 用 `/flow-next:impl-review` 审查

**不需要 stage-harness**。全程用 Flow-Next + Superpowers 的原生能力。

**用户价值**：验证"AI 能不能从模糊需求推进到代码交付"这一核心假设。

### MVP-1：加入人工决策收束（轻量 stage-harness）

**目标**：在 MVP-0 的基础上，把关键决策点收束起来。

**新增**：
1. Decision Bundle 机制（含 must_confirm / assumable / deferrable 分类）
2. PLAN→EXECUTE 门禁（人必须确认计划后才能开工）
3. `.harness/` 目录存放治理元数据

### MVP-2：加入质量保障

**新增**：
1. Checklist 式议会（单 LLM 多视角，成本低）
2. FIX 轮次限制
3. DONE 阶段交付包生成

### MVP-3：加入自治 + 深度治理

**新增**：
1. Ralph 自治模式
2. 动态强度控制
3. 真实多 reviewer 议会
4. ECC 安全扫描
5. 学习沉淀流程

## 六、总结

### V4 对原始诉求的覆盖率

| 原始诉求 | 覆盖度 | 说明 |
|---------|-------|------|
| P0-1：处理模糊需求输入 | ✅ 90% | 缺少入口标准化 |
| P0-2：自动做需求澄清 | ⚠️ 70% | **缺少决策分类机制**（哪些可假设推进） |
| P0-3：人工介入点极少 | ⚠️ 60% | **完整路径介入点过多**，需要更激进的自动放行 |
| P0-4：推进到交付物 | ✅ 85% | 缺少交付介质 checklist |
| P1-1：开箱即用 | ⚠️ 50% | **stage-harness 需要大量开发**，建议增加 MVP-0 |
| P1-2：agent/skills 方向 | ✅ 100% | 完全契合 |
| P1-3：多阶段闭环 | ✅ 95% | 覆盖良好 |
| 隐含-1：判断哪些问题值得问 | ❌ 30% | **最大 gap** |
| 隐含-2：能执行不只建议 | ✅ 100% | Flow-Next worker 直接写代码 |
| 隐含-3：多阶段协作 | ✅ 100% | 20 agent + 多阶段 |
