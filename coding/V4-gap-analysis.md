# V4 方案 vs 原始诉求 — 差距分析（修订版）

> 基于 V4 最终版（一体化插件方案）逐条对照原始诉求，识别遗漏、补强点和确认已覆盖项。

---

## 一、原始诉求一句话

> 我想要一套尽量开箱即用的 AI 能力体系，在接收到售前或客户的模糊需求后，先自动完成需求澄清，再继续推进方案、开发、测试和交付；整个过程中只把极少数关键决策点交给人，其余尽量自主完成，并最终输出可用成果。

---

## 二、P0 核心诉求逐条对照

### P0-1：能处理模糊需求输入

**原始诉求**：输入不是标准工单，而是售前/客户给出的不完整、有歧义、信息缺失的描述。

**V4 覆盖情况**：✅ 已覆盖

- `/harness:start {模糊需求}` 直接接受任意文本输入
- 内部自动做入口格式化（整理为一句话目标 + 已知约束 + 期望交付物）
- CLARIFY 阶段的 brainstorming + interview 专门处理模糊性

**无需补充。**

---

### P0-2：能自动做需求澄清

**原始诉求**：
- 识别需求中的不明确点
- 自动拆分出：必须确认 / 可假设推进 / 可延后确认
- 尽量减少需要人工确认的问题数量
- 把必须问人的问题压缩到极少数

**V4 覆盖情况**：✅ 已覆盖

- 决策分类机制（must_confirm / assumable / deferrable）直接对应这条诉求
- brainstorming 识别问题空间，interview 做收敛精炼
- assumable 项 AI 自主假设推进并记录依据，不阻断流程

**无需补充。**

---

### P0-3：人工介入点极少

**原始诉求**：
- 大部分过程 AI 自主完成
- 只有少量高价值必须拍板的点交给人
- 人的角色是审批者/决策者/兜底者

**V4 覆盖情况**：⚠️ 基本覆盖，但需要补充一个关键澄清

**已覆盖**：
- 自动放行规则：议会 PASS 自动推进，assumable 自动推进
- 低风险 feature 可全程自治（Ralph 模式）
- 只有 must_confirm 的 Decision Bundle 才暂停

**需要补充**：❌ **V4 没有明确回答一个关键问题——典型场景下到底会暂停几次？**

用户最关心的不是"有哪些机制保障少打断"，而是**"跑一个中等复杂度的 feature，我到底会被打断几次？"**

逐条分析 V4 完整路径中的潜在暂停点：

| 节点 | 是否暂停 | 条件 |
|------|---------|------|
| CLARIFY → SPEC | **视情况** | 如果 Decision Bundle 全部 assumable → 自动放行，不暂停 |
| SPEC 轻议会 | **否** | verdict = GO 自动放行 |
| SPEC → PLAN | **视情况** | 同上 |
| PLAN 计划议会 | **否** | verdict = READY 自动放行 |
| PLAN → EXECUTE | **可能暂停** | 如果有 must_confirm 项则暂停（这是最可能的暂停点） |
| EXECUTE 过程 | **否**（低/中风险） | 自治或 worker 自动推进 |
| VERIFY 验收议会 | **否** | verdict = PASS 自动放行 |
| VERIFY → DONE | **可能暂停** | Decision Bundle 通常有 must_confirm（发布确认） |
| DONE 发布议会 | **否** | verdict = RELEASE_READY 自动放行 |

**典型中等复杂度 feature 的暂停次数预估**：

| 风险等级 | 预估暂停次数 | 暂停位置 |
|---------|-----------|---------|
| 低风险 | **0~1 次** | 可能在 VERIFY→DONE 暂停一次做最终确认 |
| 中风险 | **1~2 次** | PLAN→EXECUTE 一次 + VERIFY→DONE 一次 |
| 高风险 | **2~3 次** | CLARIFY 一次 + PLAN→EXECUTE 一次 + VERIFY→DONE 一次 |

**补充建议**：在 V4 文档中增加这个**暂停次数预估表**，让用户对"极少数关键决策点"有量化预期。

### P0-4：能一路推进到交付物

**原始诉求**：
- 最终要有可执行方案、代码/系统、测试结果、可交付介质
- 不是停在分析，而是真正交付

**V4 覆盖情况**：⚠️ 基本覆盖，但"交付物"的定义需要更具体

**已覆盖**：
- `/harness:work` 产出代码 + 测试 + git 原子提交
- Evidence 记录 commits / tests / PRs
- DONE 阶段产出 release-notes.md + delivery-summary.md

**需要补充**：⚠️ **V4 的"交付物"偏向代码侧，缺少非代码交付物的考虑**

原始诉求提到的交付物包括"可执行方案、原型/流程、代码/系统、测试结果、可交付介质"。V4 目前只覆盖了代码/测试/文档。

在实际售前场景中，交付物可能是：
- 技术方案文档（给客户看的）
- 原型演示
- API 文档
- 部署指南
- 测试报告（含覆盖率数据）

**补充建议**：在 DONE 阶段增加一个**交付物 checklist**，根据需求类型动态决定需要产出什么：

```markdown
## Delivery Checklist
- [x] 源代码（已提交）
- [x] 测试用例（已通过）
- [x] Release Notes
- [ ] API 文档（如有 API 变更）
- [ ] 部署指南（如有部署变更）
- [ ] 技术方案文档（如客户需要）
```

这不需要新增机制，只需在 `delivery-summary.md` 模板中加入 checklist section。

## 三、P1 诉求逐条对照

### P1-1：尽量直接可用，少二次开发

**原始诉求**：希望工具尽量开箱即用。

**V4 覆盖情况**：⚠️ 需要坦诚评估

V4 是"一个插件"，但这个插件本身**需要大量开发工作**才能存在。

用户面临的现实是：
- **Day 1**：stage-harness 插件还不存在，需要先开发
- **开发完成后**：一个插件安装即用

这与"尽量直接可用"存在时间差。

**补充建议**：增加一个**分阶段可用策略**：

| 阶段 | 可用状态 | 用户体验 |
|------|---------|---------|
| **开发前（现在）** | 直接使用 Flow-Next + Superpowers 两个现成插件 | 能跑通 plan → work → review，但没有 Decision Bundle 和议会 |
| **MVP-1 完成后** | 安装 stage-harness 一个插件 | 完整闭环 + 议会 + Decision Bundle |
| **MVP-2 完成后** | 同上，能力增强 | + 自治 + 安全 + 交付 |

也就是说，**在 stage-harness 开发期间，用户可以先用 Flow-Next + Superpowers 体验核心价值**。stage-harness 成熟后再迁移到一个插件方案。这不是"装两个插件"的目标态，而是**开发期的临时过渡策略**。

**V4 覆盖情况**：✅ 完全契合

**无需补充。**

### P1-3：能把多个阶段串起来形成闭环

**V4 覆盖情况**：✅ 已覆盖

`/harness:start` 一条命令串联 CLARIFY → SPEC → PLAN → EXECUTE → VERIFY → FIX → DONE。

**无需补充。**

## 四、隐含约束逐条对照

### 隐含-1：不能只会问问题，还要会判断"哪些问题值得问"

**原始诉求**：AI 需要能区分必须确认的 / 可假设推进的 / 可后置确认的。减少人工确认次数而不是增加。

**V4 覆盖情况**：✅ 已覆盖

决策分类机制（must_confirm / assumable / deferrable）直接回应此诉求。

**但有一个深层问题值得思考**：⚠️ **AI 凭什么判断一个决策是 assumable 还是 must_confirm？**

V4 描述了分类机制，但没有给出**分类判断的标准/规则**。这意味着分类质量完全取决于 LLM 的 prompt 设计和推理能力。

**补充建议**：在 `decision-bundle/SKILL.md` 中定义**分类判断规则**：

| 条件 | 分类 | 依据 |
|------|------|------|
| 涉及业务方向选择（to B / to C / 定价模型 / 目标客群） | must_confirm | 业务决策不能由 AI 代拍 |
| 涉及合规/法律/安全红线 | must_confirm | 风险不可逆 |
| 有行业惯例 + 项目上下文支撑 + 后续可纠正 | assumable | 可假设推进 |
| 属于实现细节 + 不影响架构 + 后续可改 | deferrable | 不阻塞主线 |
| 不确定分类时 | must_confirm | **宁可多问一次也不偷偷拍板** |

最后一条是安全网——当 AI 对分类没有信心时，默认归为 must_confirm。

**V4 覆盖情况**：✅ 已覆盖

worker subagent 直接写代码、写测试、做原子提交。

**无需补充。**

### 隐含-3：需要多阶段协作能力

**V4 覆盖情况**：✅ 已覆盖

18 个 agent + 8 阶段状态机 + 4 类议会。

**无需补充。**

### 隐含-4：结果需要稳定、可复用

**原始诉求**：希望形成固定方法、可复用流程、可复用技能，最终交给其他人也能用。

**V4 覆盖情况**：⚠️ 部分覆盖

**已覆盖**：
- 三层经验沉淀（项目级 → 团队级 → 个人级）
- DONE 阶段 learning-candidates.md 经验提名
- memory 系统（pitfalls / conventions / decisions）

**需要补充**：❌ **"交给其他人也能用"——V4 缺少团队 onboarding 和能力复制的设计**

"可复用"不只是"经验可沉淀"，还包括：
1. **新成员如何上手？** 装了 stage-harness 后第一步做什么？
2. **团队经验如何自动加载？** 新成员自动获得团队积累的 conventions / pitfalls
3. **不同项目之间如何复用？** 项目 A 的经验如何迁移到项目 B

**补充建议**：

1. **Quick Start Guide**（docs/quick-start.md）——用一个具体例子走一遍全流程
2. **团队配置继承**——`.harness/config.json` 支持继承团队级配置（从 git 仓库或共享位置加载）
3. **经验迁移命令**——类似 `flowctl memory export/import`，支持跨项目经验包迁移

## 五、发现的新遗漏

### 5.1 ❌ 缺少"上下文窗口管理"设计

V4 的全流程涉及大量产物（epic spec、task specs、code、review reports、council verdicts...）。当流程走到 VERIFY 阶段时，上下文窗口可能已经无法容纳所有相关信息。

V2 的 DEVELOPMENT-PLAN.md 专门设计了**上下文窗口管理策略**（每个阶段只加载当前相关 artifacts），但 V4 没有继承这个设计。

Flow-Next 用 **re-anchor** 解决了 EXECUTE 阶段的上下文问题（每个 task 前重读 spec + 状态），但其他阶段没有类似机制。

**补充建议**：在 V4 中增加**分阶段上下文加载策略**：

| 阶段 | 加载 | 不加载 |
|------|------|--------|
| CLARIFY | 原始需求文本 | 无 |
| SPEC | clarification-notes + decision-bundle | 上游对话 |
| PLAN | epic spec（specs/*.md） | 原始需求、CLARIFY 细节 |
| EXECUTE | 当前 task spec + epic spec 摘要 | 其他 task、议会报告 |
| VERIFY | epic spec + 全量 diff + task evidence | 执行过程细节 |
| DONE | verification.json + delivery-summary 模板 | 执行过程 |

**每个阶段的输入是上一阶段的 artifacts，而非完整对话上下文。** re-anchor 机制应从 EXECUTE 扩展到所有阶段。

---

### 5.2 ⚠️ "测试"在 V4 中偏弱

原始诉求明确提到"测试验证"是端到端流程的一部分。V4 的 EXECUTE 阶段提到"worker 写测试 + 原子提交"，VERIFY 阶段有 `test-report.md`，但整体上测试设计比较薄弱：

1. **没有明确的测试策略前置**——V4 的 epic spec 模板里有 `## Test Strategy` section，但没有说明如何确保这个 section 足够具体
2. **没有测试覆盖率门禁**——VERIFY 阶段检查"测试质量"，但没有量化标准（例如覆盖率 > 80%）
3. **没有区分测试类型**——单元测试、集成测试、E2E 测试分别在什么时机运行？

**补充建议**：

在 PLAN 阶段的 epic spec 模板 `## Test Strategy` section 中明确要求：

```markdown
## Test Strategy
### 单元测试
- 覆盖哪些模块
- 预期覆盖率
### 集成测试
- 测试哪些接口交互
### E2E 测试（如适用）
- 测试哪些关键用户路径
### 验收标准
- 覆盖率门禁（建议 ≥ 80%）
- 所有测试必须通过
```

在 VERIFY 阶段的验收议会中增加 `test-reviewer` 的检查项：
- 测试覆盖率是否达标
- 是否有关键路径未被测试覆盖
- 测试是否稳定（非 flaky）

---

### 5.3 ⚠️ 缺少"恢复与接力"机制

原始诉求没有直接提到这一点，但在实际使用中很关键：

- Claude Code session 可能中途断开（token 耗尽、网络中断、手动退出）
- 下次打开时，需要能从上次断点继续

V4 的 `state.json` 记录了当前阶段，flowctl 记录了 task 状态，但没有明确：

1. **Session 断开后如何恢复？** 是自动检测 `.harness/` 并恢复，还是需要用户手动运行命令？
2. **多 session 间如何接力？** 前一个 session 的工作结果如何无损传递给下一个 session？

**补充建议**：

- `stage-init.sh`（SessionStart hook）应自动检测 `.harness/features/` 是否有进行中的 epic，如果有则提示用户当前进度并询问是否继续
- 每个阶段完成时自动保存 **handoff 摘要**（当前进度 + 下一步做什么 + 关键上下文），存储在 `.harness/features/{epic-id}/handoff.md`
- 新 session 启动时读取 handoff.md 恢复上下文

这就是 V2 harness.md 中提到的 "Session Handoff" 设计，V4 应该内化这个能力。

---

### 5.4 ⚠️ 缺少"回滚"的具体设计

V4 说"可回退"，但没有具体描述：

1. 回退到上一阶段时，已产出的 artifacts 怎么处理？保留还是删除？
2. 代码已提交的情况下，回退到 PLAN 意味着什么？是 git revert 还是新分支？
3. 回退后，议会 verdict 是否需要重新运行？

**补充建议**：

| 回退场景 | 产物处理 | 代码处理 | 议会处理 |
|---------|---------|---------|---------|
| SPEC → CLARIFY | 保留旧 spec 备查，重新生成 | 无代码影响 | 轻议会需重跑 |
| PLAN → SPEC | 保留旧 tasks 备查，重新生成 | 无代码影响 | 轻议会 + 计划议会需重跑 |
| EXECUTE → PLAN | 保留已完成的 task evidence | 从最后成功提交创建新分支 | 计划议会需重跑 |
| VERIFY → EXECUTE | 保留 review 报告，标记需修复项 | 在同一分支继续修复 | 验收议会需重跑 |

## 八、建议的补充项优先级

| 优先级 | 补充项 | 工作量 | 影响 |
|-------|-------|-------|------|
| **P1** | 决策分类判断规则（哪些场景 must_confirm / assumable） | 低——写入 SKILL.md 即可 | 直接影响"少打断"的核心体验 |
| **P1** | Session 恢复 / Handoff 机制 | 中——实现 handoff.md 自动生成 + SessionStart 恢复 | 影响实际可用性 |
| **P1** | 上下文窗口管理策略 | 低——在各阶段 SKILL.md 中定义加载范围 | 影响长流程稳定性 |
| **P2** | 暂停次数预估表 | 极低——文档补充 | 帮助用户建立预期 |
| **P2** | 交付物 checklist（含非代码交付物） | 低——模板补充 | 覆盖更多交付场景 |
| **P2** | 测试策略强化（覆盖率门禁 + 测试分层） | 低——模板 + reviewer 检查项 | 提升交付质量 |
| **P2** | 回滚具体设计 | 中——需要与 git 操作协调 | 影响异常恢复能力 |
| **P2** | 开发前过渡策略（先用现成插件） | 极低——文档说明 | 减少用户等待期 |
| **P3** | 团队 onboarding + 跨项目经验迁移 | 中——需要配置继承 + 导入导出 | 影响团队推广 |

## 九、总结

V4 的一体化插件设计对原始诉求的覆盖度整体**很高**（85%+）。最核心的五条 P0 诉求（模糊输入 / 需求澄清 / 少人介入 / 端到端交付 / 一个插件）都已覆盖。

**最需要补充的 3 项**（影响实际可用性）：

1. **决策分类判断规则**——让 AI 知道什么时候该假设推进、什么时候必须问人。没有这个规则，决策分类机制就是空壳。
2. **Session 恢复与 Handoff**——长流程必然跨 session，断了能接。
