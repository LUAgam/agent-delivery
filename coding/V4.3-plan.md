# Claude Code 阶段化 AI Harness V4.3 — 通用核心版（适配所有项目）

> 在 V4.2 基础上，正式做四项修订：
>
> 1. 将文档从“偏系统项目写法”调整为“通用核心 + 项目类型适配层”
> 2. 在 CLARIFY 阶段新增独立步骤：**产品/领域影响扫描**
> 3. 将需求澄清阶段定义为**Lead 主导 + 多角色可选并行**的 agent 协作模型
> 4. 将“未知问题发现机制”和“对抗式验证机制”改写为**适配所有项目的通用问题维度**
>
> 本版目标不是服务某一类系统，而是定义一套可复用于**应用、前端、后端、SDK、数据链路、基础设施、插件、工具链、文档自动化**等各类项目的统一阶段化 AI Harness。
>
> 本文档作为后续实施的唯一权威基线。

## 一、文档定位

本文档是 V4.2 的通用化修订版。

核心决策保持不变：

> **stage-harness 是一个一体化的 Claude Code 插件。**

用户只安装这一个插件，即可获得从模糊需求到可用交付物的端到端闭环能力。

stage-harness 在设计与实现层面，继续参考复用以下成熟组件的思路与代码：

## 二、最终主张

> **stage-harness** 是一个自包含的 Claude Code 插件，在接收到模糊需求后，自动完成需求澄清、规格定义、计划生成、开发执行、验证审查、问题修复、交付沉淀的全流程推进。整个过程中只把极少数关键决策点交给人，其余全部自主完成。

核心特征：

1. **一个插件满足核心主链诉求**
2. **接住模糊输入**
3. **AI 自主判断**
4. **端到端交付**
5. **多角色议会**
6. **自治运行**
7. **运行时 harness**
8. **技能进化**
9. **最少打断**
10. **项目画像驱动**
11. **分阶段澄清**
12. **未知问题主动发现**
13. **问题覆盖闭环**
14. **问题模式沉淀**

## 三、顶层设计原则

### 3.1 主状态机固定，适配层可变

主状态机保持不变：

`IDEA → CLARIFY → SPEC → PLAN → EXECUTE → VERIFY → FIX → DONE`

项目类型、风险等级、仓库数量、产物类型，只影响：

- 检查重点
- agent 并行度
- 证据形态
- 议会规模
- harness 强度

**不改变主状态机。**

### 3.2 先明确“要做什么”，再明确“会影响什么”，最后才明确“在哪里怎么做”

需求处理顺序固定为：

1. **需求本体澄清**
2. **产品/领域影响扫描**
3. **项目承载面缩圈**
4. **定点深挖**
5. **规格固化**
6. **计划与执行**

这意味着：

- 不能一上来扎进代码
- 不能一开始就让仓库现状绑架目标
- 也不能只复述需求文本而不主动找隐含影响

### 3.3 未知问题应尽量前移发现

原则上：

- **业务语义、状态变化、边界场景、性能风险、兼容风险**  
  应优先在 **CLARIFY** 阶段被发现和提出
- **实现方式、任务拆分、验证策略、证据路径**  
  应在 **PLAN** 阶段被固化
- **实现正确性、覆盖充分性、对抗场景稳定性**  
  应在 **VERIFY** 阶段被兜底

## 四、不可替换的通用机制层

以下机制是整个体系的骨架，不随项目类型变化。

### 4.1 PLAN 是后半程控制中心

后半程主链：

`PLAN → EXECUTE → VERIFY → FIX → DONE`

硬规则：

- EXECUTE 发现切分失效 → 必须回流 PLAN
- EXECUTE 发现依赖不成立 → 必须回流 PLAN
- 运行时 harness 判定任务漂移或反复失败 → 必须回流 PLAN
- 不允许“计划之外的临场扩写”静默混入

### 4.2 Decision Bundle + 决策分类

在关键阶段出口收束决策。AI 主动判断哪些可先假设推进，只把真正无法自主决策的问题交给人。

| 分类 | 处理方式 | 说明 |
|---|---|---|
| **must_confirm** | 阻断，等人拍板 | 业务方向、不可逆选型、安全/合规红线 |
| **assumable** | AI 假设推进，记录依据，不阻断 | 有足够上下文支撑，且后续可纠正 |
| **deferrable** | 不阻塞当前阶段，标记后续确认 | 不影响当前阶段闭环 |

自动放行规则：

- Decision Bundle 全部为 assumable / deferrable → 自动放行
- Decision Bundle 有 must_confirm → 阻断
- 议会 verdict = GO / PASS / READY / RELEASE_READY → 自动放行
- 议会 verdict = REVISE / HOLD / BLOCK → 阻断

### 4.3 中断预算与 Decision Packet（硬约束）

“尽量少打断人”不再只是原则，而是运行时必须遵守的硬约束。

每个 epic 在 CLARIFY 完成后即写入 `interrupt-budget.json`：

| 风险等级 | 强制中断预算 | 默认允许中断点 |
|---|---:|---|
| 低风险 | 1 次 | VERIFY→DONE |
| 中风险 | 2 次 | PLAN→EXECUTE、VERIFY→DONE |
| 高风险 | 3 次 | CLARIFY、PLAN→EXECUTE、VERIFY→DONE |

规则：

- 预算只统计 must_confirm 阻断式中断
- assumable / deferrable 不计入预算
- 超出预算后，除非触发安全/合规/业务方向/不可逆选型，否则必须转为 assumable 或 deferrable
- 同阶段禁止多次零碎追问；必须合并成一个 **Decision Packet**

Decision Packet 必须附带：

- 为什么现在必须问
- 不回答时默认怎么做
- 最晚何时需要答案

### 4.4 分层议会

| 议会类型 | 阶段 | 规模 | 关注重点 |
|---|---|---:|---|
| 轻议会 | SPEC | 3~5 | 漏项、隐含假设 |
| 计划议会 | PLAN | 5~7 | 覆盖、切分、依赖、风险 |
| 验收议会 | VERIFY | 5~7 | 实现质量、安全、测试、影响范围 |
| 发布议会 | DONE | 3~4 | 交付完整性、安全签署、治理沉淀 |

### 4.5 阶段门禁 + 可审查 / 可阻断 / 可回退

每个阶段出口必须有结构化检查，不合格则阻断，出问题可回退。

### 4.6 运行时 Harness

stage-harness 不只在阶段出口做检查，还要在执行过程中持续控偏。

运行时 harness 的硬职责：

- **Preflight**：任务开跑前校验输入上下文、依赖、工作区、预算
- **In-loop Eval**：任务执行中做漂移检测、失败计数、证据完整性检查
- **Task Smoke**：每个 task 完成后做最小可运行验证
- **Stage Smoke**：每个阶段出口做阶段级烟测
- **Auto-Diagnose**：失败时自动收集命令、日志、diff、测试结果、环境摘要
- **Rollback / Re-anchor / Re-plan**：按失败类型自动采取纠偏动作

执行中不满足以下任一条件，必须阻断并处理：

- 任务目标与 spec 不一致
- 依赖前置条件未满足
- 代码通过实现但未留下可验证 evidence
- 连续失败超过阈值
- 回归烟测失败

### 4.7 Unknowns Ledger（新增，通用问题台账）

新增统一产物：`unknowns-ledger.json`。

它不是案例列表，而是**问题台账**，按问题维度记录，不按单一需求实例记录。

作用：

- 在 CLARIFY 阶段记录“已发现但未定论”的问题
- 在 SPEC 阶段把这些问题固化为边界、约束、验收或风险
- 在 PLAN 阶段映射到 task、验证手段和证据位置
- 在 EXECUTE / VERIFY 阶段承接新增 finding
- 在 DONE 阶段沉淀为问题模式

### 4.8 Coverage Matrix（新增，覆盖映射矩阵）

新增统一产物：`coverage-matrix.json` 或 `coverage-matrix.md`。

它回答四个问题：

1. 哪类问题 / 风险已经被识别
2. 它在规格里落在哪条条目
3. 它由哪个 task / 哪类实现来承接
4. 它通过什么验证 / 证据来闭环

映射关系为：

`问题维度 → 规格条目 → 实施 task → 验证手段 → 证据位置`

任何无法映射到 task 或验证的风险，都必须显性暴露，而不能默默遗漏。

## 五、项目画像与适配层

为了让框架适配所有项目，新增一个轻量层：

> **Project Profile（项目画像）**

它的作用是：

- 识别当前项目属于哪一类
- 调整检查模板与关注权重
- 不改变主状态机与核心门禁

推荐画像类型：

| 类型 | 典型例子 |
|---|---|
| **Application / Product** | Web 产品、业务系统、桌面应用 |
| **Backend Service** | API 服务、微服务、任务服务 |
| **Frontend / Client** | Web 前端、移动端、小程序 |
| **Library / SDK** | 工具库、客户端 SDK、组件库 |
| **Data / Pipeline** | DTS、ETL、数据加工、同步链路 |
| **Infra / Platform / DevOps** | 平台能力、CI/CD、运维自动化 |
| **Plugin / Tooling / Workflow** | 插件、CLI、IDE 工具、自动化流 |
| **Docs / Content Automation** | 文档系统、知识库、内容自动化 |

硬规则：

- 项目画像只影响**关注面和模板**
- 不允许因为项目类型不同而随意跳过主阶段
- 特化术语应下沉到画像适配层，而不是占据核心正文

## 七、CLARIFY 的 Agent 协作模型

### 7.1 设计原则

需求澄清阶段支持多角色协作，但不要求每个子步骤都固定对应一个独立 agent。

更合理的规则是：

> **每个阶段对应的是“角色能力”；是否拆成独立 agent，由复杂度决定。**

因此：

- 简单需求：一个 Lead 串行推进即可
- 中等需求：1 主 + 2~3 辅
- 复杂需求 / 高模糊度 / 多承载面需求：启用 agent teams 并行

### 7.2 默认角色能力

| 角色 | 主要职责 |
|---|---|
| **Lead / Orchestrator** | 维护全局上下文、决定是否并行、汇总结果、统一对外输出 |
| **Requirement Analyst** | 澄清目标、范围、成功标准、边界、必须拍板点 |
| **Impact Analyst** | 不依赖代码细节，先做产品/领域影响扫描 |
| **Challenger** | 从边界、反例、误用、极端场景专门找遗漏问题 |
| **Project Surface Router** | 结合项目承载面做缩圈，判断后续重点深挖位置 |
| **Deep Dive Specialist** | 对高风险制品、关键边界、核心依赖做定点深挖 |
| **Clarification Synthesizer** | 汇总并归类 must_confirm / assumable / deferrable |

### 7.3 并行与串行建议

**适合并行的步骤：**

- 产品/领域影响扫描
- brainstorming / 补盲
- 项目承载面缩圈
- 定点深挖（按热点拆）

**适合串行收口的步骤：**

- 需求本体澄清
- interview 收敛
- 决策分类
- Decision Bundle / Packet 输出

### 7.4 对外输出规则

无论内部是否启用 agent teams，对外都必须遵守：

- 只允许 **Lead agent** 统一收口
- 不允许多个 agent 分别向用户提问
- 不允许不同 agent 给出冲突结论而不汇总消歧
- 所有 must_confirm 必须压缩进一个统一 Decision Packet

## 九、命令体系

| 命令 | 功能 | 阶段 |
|---|---|---|
| `/harness:start {模糊需求}` | 从模糊需求启动全流程 | 全流程 |
| `/harness:clarify {模糊需求}` | 需求澄清：本体澄清 + 影响扫描 + 收敛 | CLARIFY |
| `/harness:spec {epic-id}` | 规格定义 | SPEC |
| `/harness:plan {epic-id}` | 计划生成 | PLAN |
| `/harness:work {epic-id 或 task-id}` | 开发执行 | EXECUTE |
| `/harness:review {epic-id}` | 审查与验收 | VERIFY |
| `/harness:auto {epic-id}` | 自治模式 | 全流程自治 |
| `/harness:status` | 查看阶段进度、预算、健康度、当前命中承载面 | 任意 |
| `/harness:done {epic-id}` | 交付、沉淀、技能进化 | DONE |

## 十一、各阶段设计

### 11.0 需求入口标准化

模糊需求作为 `/harness:start` 或 `/harness:clarify` 的输入。插件内部自动做格式化：

1. 一句话目标
2. 已知约束
3. 期望交付物
4. 已知不确定项

信息不足时标记为“待澄清”。

### 11.1 PROJECT_PROFILE — 项目画像识别

这是 CLARIFY 的前置轻步骤，用于确定项目类型和默认检查模板。

| 项目 | 内容 |
|---|---|
| **工作流** | 1. 判断项目类型 → 2. 判断主要承载面 → 3. 判断风险等级 → 4. 选择默认检查模板 |
| **产物** | `project-profile.yaml` |
| **出口** | 已明确项目画像，不改变主状态机，只影响后续权重 |

### 11.2 CLARIFY — 需求澄清（增强版）

CLARIFY 不仅负责澄清用户显式需求，还必须从功能行为、状态变化、正确性、性能资源、兼容演进、安全权限、可观测性、用户体验等维度，主动扫描该需求在现有产品或项目中的后续影响，并将发现项纳入决策分类。

| 项目 | 内容 |
|---|---|
| **工作流** | 1. 需求本体澄清 → 2. 产品/领域影响扫描 → 3. brainstorming 补盲 → 4. interview 收敛 → 5. 项目承载面缩圈 → 6. 必要时定点深挖 → 7. 决策分类 → 8. 中断压缩 → 9. Decision Bundle / Packet |
| **产物** | `clarification-notes.md`、`impact-scan.md`、`unknowns-ledger.json`、`decision-bundle.json`、`decision-packet.json`、`interrupt-budget.json`、`surface-routing.json`（如已引入承载面分析） |
| **出口** | must_confirm 已压缩进统一 Packet；assumable 已记录依据；deferrable 已排入后续；后续承载面分析的重点已明确 |

#### 11.2.1 CLARIFY 的固定动作

1. **需求本体澄清**：明确目标、成功标准、生效范围、边界、必须拍板点。
2. **产品/领域影响扫描**：不深入看代码，只基于项目/产品的大体定位与模糊需求，扫描可能受影响的能力面与风险面。
3. **brainstorming 补盲**：从不同角色与视角补盲，而不只是复述需求文本。
4. **interview 收敛**：把问题整理成 must_confirm / assumable / deferrable，尽量批处理而不是零碎打断。
5. **项目承载面缩圈**：结合当前项目的实际承载面，缩小真正需要深挖的范围。
6. **定点深挖**：只对高风险制品、关键边界、核心依赖、关键实现点做定向分析，而不是全量硬扫。

#### 11.2.2 新增独立步骤：产品/领域影响扫描

这是本版新增的关键步骤，位置固定在：

> **需求本体澄清之后，项目承载面分析之前**

作用：

- 不依赖代码细节，先基于“这是什么项目 / 产品”判断可能受影响的方面
- 先形成“后续必须重点分析的影响面清单”
- 避免直接陷入代码细节而漏掉关键问题

固定输出：

1. 当前项目/产品的大体定位
2. 这次需求可能影响的主要方面
3. 高优先级待细化问题
4. 后续项目承载面分析指引

#### 11.2.3 项目承载面缩圈规则

“项目承载面”不等于“代码仓库”，它指后续真正需要分析的落点，例如：

- 代码仓库
- 关键目录
- 配置文件
- 文档
- 接口定义
- 设计稿
- 自动化流程
- 脚本 / 模板 / 数据资产

硬规则：

- 默认先做影响扫描，再做承载面缩圈
- 禁止在 CLARIFY 早期无差别全量扫描全部承载面
- 先看索引、入口、摘要、目录树、定义，再决定是否深挖正文
- 深读只针对已命中的高风险点

#### 11.2.4 决策分类判断规则

| 条件 | 分类 |
|---|---|
| 业务方向选择、目标用户、不可逆选型 | must_confirm |
| 安全、合规、权限红线 | must_confirm |
| 有行业惯例、上下文支撑、后续可纠正 | assumable |
| 不阻塞当前阶段、可后置确认 | deferrable |
| AI 对分类无把握 | must_confirm |

#### 11.2.5 CLARIFY 的 Agent 协作模式

默认由 **Lead agent** 主导，可按需调用以下角色能力：

- Requirement Analyst
- Impact Analyst
- Challenger
- Project Surface Router
- Deep Dive Specialist
- Clarification Synthesizer

复杂需求下可启用 agent teams，并行完成：

- 影响扫描
- 问题补盲
- 承载面缩圈
- 定点深挖

但所有对外澄清与决策输出必须统一收口。

#### 11.2.6 CLARIFY 的中断压缩规则

- 先尽最大可能产出 assumable 与 deferrable
- 再将剩余 must_confirm 压缩进一个 Packet
- 不允许边问边补、连续追问、同义问题拆开发
- 如果问题数量超过 5 个，必须再次做优先级压缩，只保留当前阶段真正阻断的事项

### 11.3 SPEC — 规格定义

| 项目 | 内容 |
|---|---|
| **工作流** | 1. epic 创建 → 2. 写入规格 → 3. interview 精炼 → 4. 轻议会审查 → 5. Decision Bundle |
| **产物** | `specs/{epic-id}.md`、`decision-bundle.json` |
| **出口** | 轻议会 verdict = GO，且 Decision Bundle 已处理 |

#### 11.3.1 SPEC 模板（增强版）

```markdown
# {Epic Title}

## Goal & Context
## Project Profile
## User / Operator Perspective
## State & Lifecycle Semantics
## Input / Output Contracts
## Correctness Invariants
## Conflict / Re-entry Policy
## Non-functional Budget
## Compatibility / Evolution / Recovery
## Edge Cases & Constraints
## Acceptance Criteria
## Boundaries
## Test Strategy
## Risk Register
```

要求：

- 必须把 CLARIFY 中发现的重要问题转写为规格条目
- 不能只写功能点，必须写语义、不变量、边界、风险和验收
- 不同项目类型可调整模板权重，但不可缺失上述核心维度

### 11.4 PLAN — 实施计划

| 项目 | 内容 |
|---|---|
| **工作流** | 1. 项目承载面缩圈复核 → 2. 并行 scouts 调研 → 3. 生成 task 图谱 → 4. 构建 coverage matrix → 5. 技术 plan-review → 6. 计划议会 → 7. Decision Bundle / Packet |
| **产物** | `tasks/*.json + *.md`、`coverage-matrix.json`、`plan-council-verdict.json`、`decision-packet.json`、`surface-routing.json`、`dependency-graph.json` |
| **出口** | plan-review 通过 + 计划议会 verdict ≠ BLOCK + Decision Bundle 已处理 |

PLAN 的硬要求：

- 必须把 CLARIFY / SPEC 中发现的问题映射到 task、验证和证据
- 任务拆分必须标注归属承载面
- 跨承载面 task 必须显式写明边界、依赖、联调点、回归点
- 不能只有功能拆分，没有风险与验证拆分

### 11.5 EXECUTE — 开发执行

| 项目 | 内容 |
|---|---|
| **工作流** | 1. 获取可执行 task → 2. runtime preflight → 3. worker 在 re-anchor 后实现 → 4. task smoke + evidence 校验 → 5. 原子提交 + receipt → 6. 循环直到全部完成 |
| **产物** | 代码、测试、证据、runtime receipts、原子提交 |
| **内循环** | re-anchor → preflight → 实现 → 验证 → task smoke → evidence 校验 → triage / 提交 → 下一个 task |

EXECUTE 的附加要求：

- 发现新问题时，不得顺手吞掉，必须归类为：
  - local fix
  - plan patch
  - spec patch
- 每个 task 完成前必须回答：
  - 是否暴露了新的语义、兼容或运维风险？
- 若有新增风险，必须写入 `unknowns-ledger.json` 或 finding 记录

### 11.6 VERIFY — 验证与审查（增强版）

| 项目 | 内容 |
|---|---|
| **工作流** | 1. 汇总 runtime receipts → 2. 跨模型技术 review → 3. spec compliance 审查 → 4. 安全审查 → 5. 对抗式补盲 → 6. 验收议会 → 7. stage smoke |
| **产物** | review 结果、`verification.json`、安全报告、smoke 汇总 |
| **出口** | 技术 review SHIP + 验收议会 PASS + stage smoke 通过 |

#### 11.6.1 对抗式补盲（通用版）

VERIFY 阶段新增对抗式补盲，不只验证“做了什么”，还验证“哪里可能出问题”。

输出分类：

- 已测并通过
- 未测但可接受
- 未测且必须补

### 11.7 FIX — 问题修复

review → fix 循环，最多 3 轮，超出升级人工。

FIX 阶段同样受 runtime harness 约束：

- 每轮 fix 必须明确对应的 finding
- 修复后必须重跑受影响 smoke / 测试
- 不允许“顺手额外改一批”而不更新 evidence

### 11.8 DONE — 交付与沉淀

| 项目 | 内容 |
|---|---|
| **工作流** | 1. 发布议会 → 2. 生成交付包 → 3. 经验提名 → 4. 候选 Skill 挖掘 → 5. shadow 验证 → 6. 受控晋升 |
| **产物** | `release-notes.md`、`delivery-summary.md`、`learning-candidates.md`、`candidate-skills/` |
| **出口** | 发布议会 verdict = RELEASE_READY |

DONE 阶段新增要求：

- 不只沉淀 solution pattern，还要沉淀 problem pattern
- 即：
  - 哪些做法有效
  - 哪类需求常漏哪些问题
  - 哪些问题应在 CLARIFY 更早提出
  - 哪些风险应在 PLAN 显式映射

## 十二、分层议会详细设计

### A 类：轻议会（SPEC 出口）

| 项目 | 内容 |
|---|---|
| **规模** | 3~5 reviewer |
| **角色** | 规格完整性、一致性、可落地性、可测试性、安全约束（按需） |

### B 类：计划议会（PLAN 出口）

| 项目 | 内容 |

| **核心问题** | 需求是否全覆盖？任务边界是否清晰？高风险是否有承接和验证？ |

### C 类：验收议会（VERIFY 出口）

| 项目 | 内容 |
|---|---|
| **规模** | 5~7 reviewer |
| **角色** | 代码质量、业务逻辑、安全、测试质量、影响范围、边界稳定性 |

### D 类：发布议会（DONE 出口）

| 项目 | 内容 |
|---|---|
| **规模** | 3~4 reviewer |
| **角色** | Release readiness、Security signoff、Delivery/docs、Learning governance |
| **裁决** | `RELEASE_READY` / `RELEASE_WITH_CONDITIONS` / `NOT_READY` |

## 十三、动态强度控制

根据风险等级调整流程重量，议会始终存在，只调规模与强度。

| 控制维度 | 低风险 | 中风险 | 高风险 |
|---|---|---|---|
| CLARIFY | 可跳过部分补盲 | 标准澄清 + 影响扫描 | 完整澄清 + 深度补盲 |
| Agent 并行度 | 1 主 | 1 主 + 2~3 辅 | 1 主 + 3~5 辅 |
| 轻议会 | 3 reviewer | 3 reviewer | 5 reviewer |
| Plan depth | short | standard | deep |
| 计划议会 | 3 reviewer | 5 reviewer | 7 reviewer |
| EXECUTE harness | 轻量 | 标准 | 强化 |
| 验收议会 | 3 reviewer | 5 reviewer | 7 reviewer |
| 发布议会 | 2 reviewer | 3 reviewer | 4 reviewer |
| Decision Packet | 仅关键点 | 标准 | 标准 + 额外检查点 |
| 中断预算 | 1 次 | 2 次 | 3 次 |

## 十四、自治模式

`/harness:auto {epic-id}` 启动全流程自治循环：

```text
clarify → spec → plan → [议会] → work(task1) → work(task2) → ... → review → [议会] → done
```

规则：

- 低风险 feature 可全程自治
- 中风险 feature 在关键节点暂停等人确认
- 高风险 feature 不启用完全自治
- 所有自治运行都必须受 guard hooks、runtime harness、interrupt budget 约束

自治停止条件：

- 触发 must_confirm 且预算允许中断
- 安全 / 合规 reviewer FAIL
- 同一 task 连续 3 次失败
- stage smoke 失败
- drift detector 判定偏离 spec

## 十五、经验沉淀与技能进化

### 15.1 三层沉淀模型

| 层 | 存储 | 内容 |
|---|---|---|
| 项目级 | `.harness/memory/` | 当前项目的 pitfalls / conventions / runtime observations |
| 团队级 | CLAUDE.md / rules | 团队通用工程约定 |
| 个人级 | 本地配置 | 个人偏好 |

### 15.2 自动进化 Skill 的目标

系统不是只把经验记下来，而是要识别：

- 哪些澄清动作反复出现且有效
- 哪些问题扫描维度对同类项目命中率高
- 哪些计划切分方式稳定成功
- 哪些故障排查路径可复用
- 哪些验证组合最能提前暴露问题

这些稳定模式应形成 **候选 Skill**，供后续项目复用。

### 15.3 候选 Skill 生成路径

`运行记录 / 决策 / receipts / triage / 验证结果 → 模式挖掘 → candidate-skill.md → shadow validate → auto-refine → 人工晋升`

### 15.4 自动迭代边界

为了避免“自学习污染”，自动进化必须受控：

- 只允许自动更新项目级候选 Skill
- 不允许未经人工审核直接写入团队级规则
- 低置信度模式只允许保留为 observation
- 一旦 shadow 期间出现明显误导，立即降级或废弃

## 十六、失败路径设计

| 场景 | 处理 |
|---|---|
| 需求不可行 | CLARIFY 输出 `not-feasible` + 原因，终止 |
| 技术方案不成立 | 计划议会 BLOCK，回流 SPEC 或终止 |
| 外部依赖不可用 | task 标记 `blocked` |
| Epic 被取消 | 标记关闭，保留产物 |
| Token 预算耗尽 | 暂停，保存进度 |
| 自治循环异常 | guard hooks 暂停，输出错误报告 |
| 连续 task 失败 3 次 | 自动 triage 后回流 PLAN 或升级人工 |
| stage smoke 失败 | 阻断后进入 FIX |
| 候选 Skill shadow 失败 | 降级为 observation |
| 中断预算耗尽但仍需人拍板 | 仅安全/合规/业务方向问题允许破例中断，并写入审计 |

## 十八、关键风险与缓解

| 风险 | 缓解 |
|---|---|
| 文档再次滑向某一类项目 | 核心正文只写通用机制，项目特化下沉到画像适配层 |
| 影响扫描流于空泛 | 固定输出影响面清单、高优先级待细化问题、后续分析指引 |
| agent teams 成本过高 | 默认 1 主串行，复杂场景再启用并行 |
| 议会 token 成本高 | 动态强度控制 |
| 自治模式安全风险 | guard hooks + 预算 + 人工检查点 |
| 单插件体量过大 | skill 模块化，按需加载 |
| 自动进化 Skill 产生错误模式 | 只自动更新项目级候选，团队级必须人工审核 |
| 中断预算过严导致误推进 | 安全/合规/业务方向保留破例通道 |
| 覆盖矩阵流于形式 | PLAN 阶段强制校验“问题是否映射到 task、验证、证据” |

## 十九、度量面

为了让上述机制真正可验证，必须有量化指标。

| 指标 | 含义 | 目标 |
|---|---|---|
| `interrupt_count` | 实际强制中断次数 | 不超过预算 |
| `packet_compression_ratio` | 原始 must_confirm 数 / 最终 Packet 数 | 越高越好 |
| `avoidable_interrupts` | 本可假设推进却发起中断的次数 | 趋近 0 |

## 二十二、结论

V4.3 的最终形态仍然是一个自包含的 Claude Code 插件，但其文档表达已从“偏某类系统项目”调整为“适配所有项目的通用核心版”。

本版最重要的变化不是增加更多阶段，而是把原有阶段升级为更稳定的通用机制：

1. **核心正文通用化**：正文只写所有项目都成立的机制，特化点下沉到项目画像适配层。
2. **CLARIFY 前移找坑**：在需求澄清阶段新增“产品/领域影响扫描”，先基于项目定位与模糊需求发现可能受影响的大方面，再进入承载面分析与代码细化。
3. **CLARIFY 支持多 agent 协作**：默认由 Lead 串行推进；复杂需求可让影响扫描、问题补盲、承载面缩圈、定点深挖并行执行，但必须统一收口。
4. **未知问题台账化**：通过 `unknowns-ledger` 与 `coverage-matrix`，把问题发现、规格固化、实施承接、验证闭环和经验沉淀串起来。
5. **VERIFY 通用化补盲**：不再用偏某类系统的术语写死验证项，而是用所有项目都能套用的对抗式问题维度来兜底。

这使得 stage-harness 不再只是一个偏系统工程或偏多仓项目的流程文档，而是一套真正可推广到不同项目类型的 Claude Code 阶段化 AI Harness 基线。
