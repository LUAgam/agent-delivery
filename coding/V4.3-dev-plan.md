# Stage-Harness V4.3 开发方案（开发参考手册）

> 基于 V4.3-plan.md 设计稿，结合仓库内 Flow-Next、Superpowers、ECC 三大参考实现，产出可落地的分阶段开发方案。
>
> 本文档的目标读者是"实际动手写代码的人（或 AI agent）"，每一节都会指明：做什么、怎么做、参考哪段现有代码、产出什么文件。

## 一、总体架构与技术选型

### 1.1 插件形态

Stage-harness 是一个 **Claude Code 插件**，遵循 Claude Code 插件约定：

- `.claude-plugin/plugin.json` 声明元数据
- `commands/` 存放 slash 命令（Markdown + YAML frontmatter）
- `agents/` 存放 subagent 定义
- `skills/` 存放技能包（`SKILL.md`）
- `hooks/` 存放 hooks 配置和脚本
- `scripts/` 存放 CLI 工具和辅助脚本
- `templates/` 存放结构化产物模板

**参考实现：**
- 插件 manifest 格式 → `gmickel-claude-marketplace/plugins/flow-next/.claude-plugin/plugin.json`
- 命令→技能委托模式 → Flow-Next 的 `commands/flow-next/*.md`（薄壳 + `must invoke skill X`）
- Hooks 配置 → `gmickel-claude-marketplace/plugins/flow-next/hooks/hooks.json`

### 1.2 状态管理

所有运行时状态存储在项目根目录的 `.harness/` 下，结构对标 V4.3 文档第十节。

**核心 CLI 工具：`harnessctl`**（Python，对标 Flow-Next 的 `flowctl`）

- 零外部依赖，随插件捆绑
- 通过 `${CLAUDE_PLUGIN_ROOT}/scripts/harnessctl` 路径引用
- 管理 epic/task/state/memory/config 全部状态

**参考实现：**
- `flowctl` 的架构 → `gmickel-claude-marketplace/plugins/flow-next/scripts/flowctl`（bash wrapper）+ `flowctl.py`（Python 主体）
- 状态目录 → Flow-Next 使用 `.flow/`，stage-harness 使用 `.harness/`

### 1.3 Agent 协作模型

| 模式 | 适用场景 | 实现方式 |
|------|---------|---------|
| Lead 串行 | 简单/低风险需求 | Lead agent 单独推进各步骤 |
| 1主 + N辅 | 中等需求 | Lead 通过 `Task` 工具调度 subagent |
| Agent Teams 并行 | 复杂高模糊度需求 | Lead 并行派发多个 scout/analyst |

**参考实现：**
- Worker subagent 模式 → `flow-next/agents/worker.md`（re-anchor → implement → commit → review → done）
- 并行 scout 调度 → `flow-next/skills/flow-next-plan/` 的 steps.md（并行 Task 调度 repo-scout、context-scout 等）
- Superpowers 的 spec-review subagent → `superpowers/skills/brainstorming/spec-document-reviewer-prompt.md`

### 1.4 平台约束

1. **插件 subagent 不支持 hooks/mcpServers/permissionMode frontmatter**（会被静默忽略）。需要这些字段的 agent 需在安装时下发到项目 `.claude/agents/`。
2. **hooks 通过 `hooks/hooks.json` 按约定自动加载**，不写在 `plugin.json` 内。
3. **所有 CLI 路径使用** `${DROID_PLUGIN_ROOT:-${CLAUDE_PLUGIN_ROOT}}/scripts/harnessctl`（兼容 Factory Droid）。
4. **Agent 权限用 `disallowedTools` 黑名单**（不用 `tools` 白名单，因为工具名跨平台不一致）。

## 三、`harnessctl` CLI 设计

### 3.1 架构

对标 Flow-Next 的 `flowctl`，用 Python 实现，零外部依赖。

```text
scripts/
├── harnessctl                  # #!/bin/bash → exec python3 harnessctl.py "$@"
└── harnessctl.py               # 主体，~2000-3000 行
```

### 3.2 命令体系

| 命令组 | 子命令 | 说明 | 对标 flowctl |
|--------|--------|------|-------------|
| `init` | | 初始化 .harness/ | `flowctl init` |
| `config` | `get`, `set`, `list` | 配置管理 | `flowctl config` |
| `profile` | `detect`, `set`, `show` | 项目画像管理 | — |

| `task` | `create`, `show`, `list`, `start`, `done`, `block`, `fail` | Task 管理 | `flowctl task` |
| `state` | `get`, `set`, `transition`, `history` | 状态机管理 | `flowctl state-path` |
| `surface` | `scan`, `route`, `show` | 承载面管理 | — |
| `decision` | `add`, `classify`, `bundle`, `packet` | 决策管理 | — |
| `interrupt` | `init`, `consume`, `status`, `budget` | 中断预算 | — |
| `council` | `convene`, `verdict`, `history` | 议会管理 | — |

| `handoff` | `save`, `restore` | 断点续做 | `flowctl checkpoint` |

### 3.3 状态机实现

```python
STATES = ['IDEA', 'CLARIFY', 'SPEC', 'PLAN', 'EXECUTE', 'VERIFY', 'FIX', 'DONE']

TRANSITIONS = {
    'IDEA':    ['CLARIFY'],
    'CLARIFY': ['SPEC'],
    'SPEC':    ['PLAN'],
    'PLAN':    ['EXECUTE'],
    'EXECUTE': ['VERIFY', 'PLAN'],     # 可回流 PLAN
    'VERIFY':  ['FIX', 'DONE'],
    'FIX':     ['VERIFY', 'PLAN'],     # 可回流 PLAN
    'DONE':    [],
}
```

每次状态转换写入 `.harness/features/{epic-id}/state.json`，包含：
- `current_state`
- `previous_state`
- `transition_reason`
- `timestamp`
- `gate_verdict`（门禁裁决）
- `decision_bundle_id`（关联决策包）

**Epic:**

## 四、各引擎详细设计与实现映射

### 4.1 阶段编排引擎（Orchestration Engine）

**职责：** 维护主状态机、驱动阶段门禁、管理 Decision Bundle 和中断预算。

**实现方式：**
- `skills/harness-start/SKILL.md` 作为全流程入口
- 内部通过调用 `harnessctl state transition` 驱动状态流转
- 每个阶段出口调用 `skills/stage-gate/SKILL.md` 执行门禁检查

**参考复用：**

| 机制 | 参考来源 | 复用什么 | 改造什么 |
|------|---------|---------|---------|
| Epic/Task 模型 | Flow-Next `flowctl epic/task` | 数据结构、CRUD 命令、JSON 存储 | 增加 state/surface/risk 字段 |
| 状态机转换 | Flow-Next `flowctl state-path` | 状态持久化、转换日志 | 增加门禁裁决和回流路径 |
| 断点续做 | Flow-Next `flowctl checkpoint` | 进度快照与恢复 | 增加 handoff.md 人类可读摘要 |

**关键文件：**
- `skills/harness-start/SKILL.md` — 解析模糊需求 → 格式化 → 初始化 epic → 触发 CLARIFY
- `skills/stage-gate/SKILL.md` — 门禁检查、裁决、自动放行 / 阻断逻辑
- `skills/decision-bundle/SKILL.md` — Decision Bundle 收集、分类、压缩、Packet 生成
- `skills/interrupt-budget/SKILL.md` — 中断预算初始化、消耗跟踪、超限处理

### 4.2 澄清引擎（Clarify Engine）

**职责：** 需求本体澄清 → 影响扫描 → brainstorming 补盲 → interview 收敛 → 承载面缩圈 → 定点深挖 → 决策分类 → 中断压缩

**实现方式：** 一个 `skills/clarify/SKILL.md` 编排 6 个固定动作，每个动作可通过 `Task` 工具调度相应 agent。

**参考复用：**

| 步骤 | 参考来源 | 复用什么 |
|------|---------|---------|
| 需求本体澄清 | Superpowers `brainstorming/SKILL.md` | 一次一问、多选优先、HARD-GATE 不准提前写代码、spec review loop |
| 影响扫描 | V4.3 新增，无直接参考 | 新写，但可参考 Flow-Next 的 `flow-gap-analyst` agent 的"找遗漏"模式 |
| brainstorming 补盲 | Superpowers `brainstorming/SKILL.md` | 2-3 方案对比 + 推荐 + 理由 |
| interview 收敛 | Flow-Next `flow-next-interview/SKILL.md` | 深度 Q&A 模式、结构化输出 |
| 承载面缩圈 | Flow-Next plan scouts（repo-scout 等并行调度） | 并行 Task 调度多个 scout，收口后输出 surface-routing.json |
| 定点深挖 | Flow-Next `context-scout`（RepoPrompt 深度分析） | 只对高风险点做定向分析 |

**CLARIFY 的 Skill 编排流程：**

```markdown
# CLARIFY Skill 内部流程

1. 读取 epic 的模糊需求输入
2. 调用 harnessctl profile detect → 获取项目画像
3. [串行] 需求本体澄清（Lead 直接执行）
   - 明确目标、成功标准、生效范围、边界
   - 产出：clarification-notes.md
4. [可并行] 产品/领域影响扫描
   - 简单需求：Lead 直接执行
   - 复杂需求：Task → impact-analyst agent
   - 产出：impact-scan.md
5. [可并行] brainstorming 补盲
   - Task → challenger agent（从边界、反例、误用找遗漏）
   - 产出：补充到 unknowns-ledger.json
6. [串行] interview 收敛
   - 整理 must_confirm / assumable / deferrable
   - 产出：决策分类初稿
7. [可并行] 承载面缩圈
   - Task → project-surface-router + 各 scout agent
   - 产出：surface-routing.json
8. [按需] 定点深挖
   - 仅对 surface-routing 标记的高风险点
   - Task → deep-dive-specialist agent
9. [串行] 决策分类 + 中断压缩
   - 合并所有 agent 输出
   - 压缩 must_confirm 到一个 Decision Packet
   - 写入 decision-bundle.json / decision-packet.json / interrupt-budget.json
```

**Agent 定义要点：**

`agents/impact-analyst.md`（示例）:
```yaml
---
name: impact-analyst
description: 分析需求对现有产品/项目的影响面。不深入代码细节，基于项目定位判断可能受影响的方面。
model: inherit
disallowedTools: Edit, Write, Task
color: "#F59E0B"
---
```

Body 参考 Flow-Next 的 `flow-gap-analyst.md` 写法：给出明确的分析维度、输出格式要求，限制为只读（`disallowedTools: Edit, Write, Task`）。

### 4.3 项目承载面分析引擎（Project Surface Engine）

**职责：** 识别项目画像、路由承载面、构建依赖图。

**实现方式：**

| 组件 | 实现 | 参考 |
|------|------|------|
| `project-profile/SKILL.md` | 扫描项目根目录结构 → 匹配 profile type → 写入 `.harness/project-profile.yaml` | V4.3 第五章的 8 种画像类型 |
| `project-surface/SKILL.md` | 调度 scout agents 并行扫描 → 收口输出 surface-index.json | Flow-Next 的 `flow-next-prime/SKILL.md`（9 个 scout 并行调度） |
| Scout agents | `repo-router`、`docs-scout`、`design-scout`、`config-scout`、`symbol-navigator`、`dependency-mapper` | Flow-Next 的 20 个 agents（build-scout、env-scout、testing-scout 等） |

**项目画像自动检测逻辑（`harnessctl profile detect`）：**

```python
def detect_profile(project_root):
    indicators = {}
    # 检测 package.json / tsconfig → Frontend
    # 检测 Dockerfile + go.mod / pom.xml → Backend Service
    # 检测 setup.py + pyproject.toml + 无 app 入口 → Library/SDK
    # 检测 .github/workflows + terraform → Infra/DevOps
    # 检测大量 .md + mkdocs.yml → Docs/Content
    # ... 更多启发式规则
    return best_match(indicators)
```

### 4.4 计划与执行引擎（Plan & Execute Engine）

**职责：** PLAN 阶段的 scout 调研 + task 图谱生成 + coverage matrix 构建；EXECUTE 阶段的 worker 执行循环。

#### PLAN 阶段

**参考复用：** 直接对标 Flow-Next 的 `flow-next-plan/SKILL.md`，但增强以下维度：

| 增强点 | 做法 |
|--------|------|
| coverage-matrix | PLAN 完成后强制运行 `harnessctl coverage map` 检查所有 unknowns-ledger 条目是否映射到 task |
| 跨承载面标注 | 每个 task 必须标注 `surface` 字段 |
| 风险任务拆分 | 高风险 unknowns-ledger 条目必须有对应的验证 task |

**PLAN Skill 流程：**
```
1. 承载面缩圈复核 → 读取 surface-routing.json
2. 并行 scouts 调研 → Task 调度 repo-router、docs-scout 等
3. 生成 task 图谱 → harnessctl task create（批量）
4. 构建 coverage matrix → harnessctl coverage map
5. 技术 plan-review → Task → plan-reviewer agent
6. 计划议会 → skills/council/SKILL.md（5-7 reviewer）
7. Decision Bundle / Packet → 收口输出
```

#### EXECUTE 阶段

**核心模式：Worker Subagent**，直接对标 Flow-Next 的 `worker.md`。

**Worker 执行循环：**
```
对每个 ready task:
  1. 主会话调度 Task → worker agent
  2. Worker Phase 1: Re-anchor
     - harnessctl show <task-id> --json
     - harnessctl cat <task-id>
     - harnessctl show <epic-id> --json
     - git status / git log -5
     - 读取 memory（如启用）
  3. Worker Phase 2: Preflight
     - 校验依赖前置条件
     - 校验工作区状态
     - 基线测试通过
  4. Worker Phase 3: Implement
     - 记录 BASE_COMMIT
     - 实现功能 + 编写测试
  5. Worker Phase 4: Task Smoke
     - 运行测试
     - 校验证据完整性
  6. Worker Phase 5: Commit + Receipt
     - git add -A && git commit
     - harnessctl receipt write
     - harnessctl task done
  7. 主会话校验 task status = done → 下一个 task
```

**关键差异（相比 Flow-Next worker）：**
- 增加 Phase 2 Preflight（Flow-Next 没有）
- 增加 receipt 写入（对标运行时 harness 要求）
- Task 完成前必须回答"是否暴露新风险"

### 4.5 运行时 Harness 引擎（Runtime Harness Engine）

**职责：** 执行过程中持续控偏，不只在阶段出口检查。

**实现方式：** `skills/runtime-harness/SKILL.md` 定义一组检查规则，通过 hooks 和 worker 内嵌逻辑两种方式落地。

| 检查点 | 实现方式 | 参考 |
|--------|---------|------|
| **Preflight** | Worker Phase 2 内嵌 | — |
| **In-loop Eval** | Hooks `post-tool-use` 检测漂移 | ECC 的 `scripts/hooks/quality-gate.js` |
| **Task Smoke** | Worker Phase 4 内嵌 | Flow-Next worker Phase 5（验证测试） |
| **Stage Smoke** | Stage-gate skill 内嵌 | — |
| **Auto-Diagnose** | 失败时自动收集 diff/log/env | Flow-Next 的 `flowctl done --evidence-json` |
| **Rollback/Re-anchor** | 失败 3 次回流 PLAN | Flow-Next 的 re-anchor 机制 |

**Hooks 配置（`hooks/hooks.json`）：**

```json
{
  "hooks": [
    {
      "event": "SessionStart",
      "scripts": [{ "path": "hooks/scripts/session-start.sh" }]
    },
    {
      "event": "PreToolUse",
      "scripts": [{
        "path": "hooks/scripts/pre-tool-use.sh",
```

**SessionStart hook：** 检测 `.harness/` 是否存在，如有则自动恢复上下文（读取当前 epic、阶段、进度），对标 Superpowers 的 `hooks/session-start` 注入 `using-superpowers` skill 的做法。

**Stop hook：** 自动保存 handoff 文件，对标 ECC 的 Stop hook（session summary + pattern extraction）。

### 4.6 分层议会引擎（Council Engine）

**职责：** 在 SPEC/PLAN/VERIFY/DONE 四个出口召集 reviewer agent 并行审查，汇总裁决。

**实现方式：** `skills/council/SKILL.md` 接受参数（议会类型 + epic-id），按类型调度不同数量和角色的 reviewer。

| 议会类型 | 阶段 | 并行 reviewer | 裁决选项 |
|---------|------|-------------|---------|
| 轻议会 | SPEC | 3-5 | GO / REVISE / HOLD |
| 计划议会 | PLAN | 5-7 | READY / REVISE / BLOCK |
| 验收议会 | VERIFY | 5-7 | PASS / FAIL |
| 发布议会 | DONE | 3-4 | RELEASE_READY / RELEASE_WITH_CONDITIONS / NOT_READY |

**调度模式：**

```markdown
# 议会召集流程

1. 根据议会类型确定 reviewer 列表
2. 为每个 reviewer 准备上下文（spec + evidence + 特化检查清单）
3. 并行 Task 调度所有 reviewer agent
4. 收集所有 verdict → harnessctl council verdict write
5. 汇总规则：
   - 任何 BLOCK/FAIL → 整体 BLOCK/FAIL
   - 全部 GO/PASS/READY → 整体通过
   - 混合 → 取最严格
```

**参考复用：**
- Reviewer agent 定义 → ECC 的 `agents/code-reviewer.md`、`agents/security-reviewer.md`
- 并行调度模式 → Superpowers 的 `dispatching-parallel-agents/SKILL.md`
- Verdict 格式 → ECC 的 quality-gate JSON output 模式

### 4.8 自治引擎（Auto Engine）

**职责：** `/harness:auto` 驱动全流程自治循环。

**实现方式：** 对标 Flow-Next 的 Ralph 自治循环。

| 组件 | Flow-Next Ralph | Stage-Harness Auto |
|------|----------------|-------------------|
| 循环入口 | `ralph.sh`（外部脚本） | `skills/auto/SKILL.md`（内部 skill） |
| 决策器 | `flowctl next` | `harnessctl state next`（返回下一步动作） |
| Guard hooks | `ralph-guard.py` | `hooks/scripts/pre-tool-use.sh` |
| 停止条件 | Receipt 检查 | interrupt-budget + receipt + drift detector |
| 恢复机制 | 每次迭代 fresh session | handoff.md + state.json 恢复 |

**自治循环逻辑：**
```
LOOP:
  action = harnessctl state next
  SWITCH action:
    case "clarify"  → invoke /harness:clarify
    case "spec"     → invoke /harness:spec
    case "plan"     → invoke /harness:plan
    case "work"     → invoke /harness:work (next ready task)
    case "review"   → invoke /harness:review
    case "done"     → invoke /harness:done → EXIT
    case "blocked"  → 输出阻断原因 → EXIT (等人)
  CHECK stop_conditions:
    - must_confirm 触发 && budget 允许 → EXIT
    - security/compliance FAIL → EXIT
    - 同一 task 连续 3 次失败 → EXIT
    - stage smoke 失败 → EXIT
    - drift detected → EXIT
```

### 4.9 学习与技能进化引擎

**职责：** 经验记忆 → 模式挖掘 → 候选 Skill → shadow 验证 → 受控晋升。

**参考复用：**

| 层面 | 参考来源 |
|------|---------|
| Memory 存储 | Flow-Next `flowctl memory`（pitfalls/conventions/decisions） |
| 经验提取 | ECC 的 continuous-learning（Stop hook 提取 pattern） |
| Skill 结构 | Superpowers `writing-skills/SKILL.md`（Skill 编写规范） |

**三层存储：**
```text
.harness/memory/
├── pitfalls.md          # 陷阱记录
├── conventions.md       # 约定记录
├── decisions.md         # 决策记录
├── observations/        # 低置信度观察
│   └── obs-001.json
└── candidate-skills/    # 候选技能
    └── cs-001.md
```

**自动进化边界（硬规则）：**
- 只允许自动更新 `.harness/memory/` 下的项目级内容
- 不允许自动写入 CLAUDE.md 或团队级规则
- 候选 Skill 需人工审核后才能晋升到 `skills/` 目录

## 八、开发约定与规范

### 8.1 Skill 编写规范

遵循 Superpowers 的 `writing-skills/SKILL.md` 约定：

- **Frontmatter** 只含 `name` 和 `description`，总计不超过 1024 字符
- **`description`** 写触发条件（"Use when..."），不写步骤摘要
- **Body** 包含：概述、何时触发、核心流程、快速参考、集成点、红线
- 子技能通过 `**REQUIRED SUB-SKILL:** Use harness:xxx` 引用

### 8.2 Agent 编写规范

- YAML frontmatter：`name`、`description`、`model`（默认 `inherit`）、`disallowedTools`、`color`
- 只读 agent（scout 类）：`disallowedTools: Edit, Write, Task`
- 执行 agent（worker）：`disallowedTools: Task`（禁止自行派发子任务）
- Body 给出明确的输入格式、分析维度、输出格式要求

### 8.3 CLI 编写规范

- 零外部依赖（纯 Python 标准库）
- 所有命令输出支持 `--json` 标志（便于 agent 解析）
- 错误输出到 stderr，结构化数据输出到 stdout
- 文件操作使用原子写入（先写 tmp 再 rename）

### 8.4 模板编写规范

- JSON 模板提供完整 schema 注释（在 docs/ 中描述）
- Markdown 模板使用 `{placeholder}` 标记可填充位置
- 所有模板文件可通过 `harnessctl` 实例化

## 九、风险与缓解

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| 插件 subagent 不支持 hooks frontmatter | 高 | 提供 `harnessctl setup` 命令将需要 hooks 的 agent 下发到 `.claude/agents/` |
| harnessctl.py 代码量过大（>2000行） | 中 | 拆分为 `lib/` 目录下的多个模块，主入口只做 dispatch |
| 议会 token 成本过高 | 高 | 动态强度控制：低风险 3 reviewer、高风险 7 reviewer；reviewer prompt 控制回复长度 |
| Agent 输出格式不稳定 | 中 | 在 agent prompt 中明确 JSON schema 要求；harnessctl 做容错解析 |
| 自治循环失控 | 高 | 三重防线：interrupt-budget 硬约束 + guard hooks + 连续失败 3 次自动停止 |
| 状态文件损坏 | 中 | `harnessctl validate` 做完整性校验；关键操作前自动备份 |
| 上下文窗口溢出 | 高 | 每阶段只加载上一阶段 artifacts（不加载完整历史）；先摘要后定点；re-anchor 控制读取范围 |
