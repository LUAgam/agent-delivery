## 二、目标系统概述

### 2.1 一句话定义

构建一个名为 `stage-harness` 的自包含 Claude Code 插件，在接收到模糊需求后，自动完成需求澄清、规格定义、计划生成、开发执行、验证审查、问题修复、交付沉淀的全流程推进，适配所有项目类型。

### 2.2 核心特征（V4.3 新增标记）

| # | 特征 | V4.3 新增/变化 |
|---|------|---------------|
| 1 | 一个插件满足核心主链诉求 | — |
| 2 | 接住模糊输入 | — |
| 3 | AI 自主判断 | ✅ 决策分类 + 中断预算硬约束 |
| 4 | 端到端交付 | — |
| 5 | 多角色议会 | ✅ 四类分层议会 |
| 6 | 自治运行 | ✅ `/harness:auto` |
| 7 | 运行时 harness | ✅ preflight → in-loop eval → task/stage smoke → auto-diagnose → rollback |
| 8 | 技能进化 | ✅ 候选 Skill → shadow validate → auto-refine → 人工晋升 |
| 9 | 最少打断 | ✅ 中断预算 + Decision Packet 硬约束 |
| 10 | **项目画像驱动** | ✅ 新增 |
| 11 | **分阶段澄清** | ✅ 新增（6 步固定动作） |
| 12 | **未知问题主动发现** | ✅ 新增（unknowns-ledger） |
| 13 | **问题覆盖闭环** | ✅ 新增（coverage-matrix） |
| 14 | **问题模式沉淀** | ✅ 新增 |

### 2.3 主状态机

```text
IDEA → CLARIFY → SPEC → PLAN → EXECUTE → VERIFY → FIX → DONE
                                      ↑        │
                                      └────────┘
```

不因项目类型、风险等级、仓库数量、产物类型而改变。

### 2.4 核心组件分工（V4.3 版）

```text
PROJECT_PROFILE → 项目画像识别（harness 内建）
CLARIFY         → Lead + 多角色 agent 协作（harness 内建 + Superpowers brainstorming 可选）
SPEC            → ShipSpec（PRD / SDD / TASKS）+ 轻议会
PLAN            → bridge + deep-plan（bridge-spec → plan / tdd / sections）+ 计划议会
EXECUTE         → deep-implement（section 级 TDD + 内置审查）+ 运行时 harness
VERIFY          → Ring（并行 reviewer 验收议会）+ 对抗式补盲
FIX             → deep-implement + Ring 重跑
DONE            → ECC 子集 + 发布议会 + 技能进化
```

## 四、组件复用与裁剪清单

### 4.1 V4.3 复用边界（与前序方案的差异）

V4.3 最大的结构性变化是：**CLARIFY 阶段从"仅复用 Superpowers brainstorming"变为"harness 内建多角色 agent 体系 + Superpowers brainstorming 可选"。**

| 组件 | V4.2 复用方式 | V4.3 复用方式 | 变化原因 |
|------|-------------|-------------|---------|
| **Superpowers** | CLARIFY 主负责 | CLARIFY 辅助（brainstorming 可选） | CLARIFY 升级为 Lead + 多角色体系，Superpowers 不再是主控 |
| **ShipSpec** | SPEC 主负责 | 不变 | — |
| **deep-plan** | PLAN 主负责 | 不变 | — |
| **deep-implement** | EXECUTE 主负责 | 不变 | — |
| **Ring** | VERIFY 主负责 | 不变 | — |
| **ECC** | DONE 治理 | 不变（+ 安全审查可前移到 VERIFY） | — |

### 4.2 组件裁剪规则

| 组件 | 保留 | 禁用 | 禁用原因 |
|------|------|------|---------|
| **Superpowers** | `brainstorming` 技能 | `writing-plans`、`executing-plans`、`subagent-driven-development`、`requesting-code-review` 的自动触发 | 后续链路与 deep-plan/deep-implement/harness 内建 agent 冲突 |
| **ShipSpec** | `/feature-planning` → PRD/SDD/TASKS | `/implement-task`、`/implement-feature`、Stop 钩子（task-loop-hook、feature-retry-hook、planning-refine-hook） | 执行主循环由 deep-implement 承担 |
| **deep-plan** | `/deep-plan` 全量 | — | 全量保留 |
| **deep-implement** | `/deep-implement` 全量 | — | 全量保留 |
| **Ring** | `default` 的 review 能力 | `pm-team`、`dev-team`、`pmo-team`、`finops-team` | 仅保留 review |
| **ECC** | `security-reviewer`、`harness-optimizer` | 其余 26 代理 | 避免职责混淆 |

## 六、命令体系设计

### 6.1 命令列表

| 命令 | 功能 | 触发阶段 | 关键输入 | 关键输出 |
|------|------|---------|---------|---------|
| `/harness:start {模糊需求}` | 从模糊需求启动全流程 | 全流程 | 模糊需求文本 | 自动推进到 CLARIFY |
| `/harness:clarify {模糊需求}` | 需求澄清全流程 | CLARIFY | 模糊需求文本 | `clarification-notes.md`、`impact-scan.md`、`unknowns-ledger.json`、`decision-bundle.json`、`decision-packet.json`、`interrupt-budget.json` |
| `/harness:spec {epic-id}` | 规格定义 | SPEC | CLARIFY 产物 | `specs/{epic-id}.md`、轻议会 verdict |
| `/harness:plan {epic-id}` | 计划生成 | PLAN | SPEC 产物 | `tasks/*.json`、`coverage-matrix.json`、计划议会 verdict |
| `/harness:work {epic-id\|task-id}` | 开发执行 | EXECUTE | PLAN 产物 | 代码、测试、evidence、runtime receipts |
| `/harness:review {epic-id}` | 审查与验收 | VERIFY | EXECUTE 产物 | `verification.json`、验收议会 verdict |
| `/harness:auto {epic-id}` | 自治模式 | 全流程自治 | CLARIFY 产物起 | 全部交付物 |
| `/harness:status` | 查看进度 | 任意 | 无 | 阶段进度、预算、健康度 |
| `/harness:done {epic-id}` | 交付沉淀 | DONE | VERIFY 产物 | `release-notes.md`、`delivery-summary.md`、候选 Skill |

### 6.2 命令实现要点

每个命令 markdown 文件需包含：

1. **角色定义**：明确 Lead agent 的行为边界
2. **前置检查**：验证当前阶段状态是否允许执行
3. **执行步骤**：按 V4.3 定义的固定动作序列
4. **产物要求**：列出必须产出的文件及其模板
5. **出口条件**：门禁规则 + Decision Bundle 处理规则
6. **错误处理**：失败路径与升级规则

## 七、各阶段详细设计

### 7.0 PROJECT_PROFILE — 项目画像识别

**定位**：CLARIFY 的前置轻步骤。

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 判断项目类型 → 2. 判断主要承载面 → 3. 判断风险等级 → 4. 选择默认检查模板 |
| **产物** | `.harness/project-profile.yaml` |
| **出口** | 已明确项目画像 |
| **实现** | harness 内建逻辑，不依赖外部组件 |

**`project-profile.yaml` 模板**：

```yaml
project_type: "backend_service"        # application|backend_service|frontend|library|data_pipeline|infra|plugin|docs
primary_surfaces:                       # 主要承载面
  - type: "code_repository"
    path: "."
  - type: "api_definition"
    path: "./api/"
risk_level: "medium"                    # low|medium|high
check_template: "backend_service"       # 对应 §20 的适配层
detected_at: "2026-03-27T10:00:00Z"
confidence: 0.85
overrides: {}                           # 人工覆盖
```

**风险等级自动评估因素**：

| 因素 | 低风险 | 中风险 | 高风险 |
|------|--------|--------|--------|
| 安全敏感度 | 不涉及 | 部分涉及 | 核心认证/授权 |
| 跨系统范围 | 单模块 | 2-3 模块 | 跨服务/跨仓库 |
| 数据变更 | 无 | 可逆变更 | 不可逆迁移 |
| 不确定性 | 需求明确 | 部分模糊 | 高度模糊 |

### 7.1 CLARIFY — 需求澄清（V4.3 增强版）

**这是 V4.3 变化最大的阶段。**

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 需求本体澄清 → 2. 产品/领域影响扫描 → 3. brainstorming 补盲 → 4. interview 收敛 → 5. 项目承载面缩圈 → 6. 必要时定点深挖 → 7. 决策分类 → 8. 中断压缩 → 9. Decision Bundle / Packet |
| **产物** | `clarification-notes.md`、`impact-scan.md`、`unknowns-ledger.json`、`decision-bundle.json`、`decision-packet.json`、`interrupt-budget.json`、`surface-routing.json` |
| **出口** | must_confirm 已压缩进统一 Packet；assumable 已记录依据；deferrable 已排入后续 |

#### 7.1.1 Agent 协作模型

| 角色 | Agent 文件 | 职责 | 触发条件 |
|------|-----------|------|---------|
| **Lead / Orchestrator** | `lead-orchestrator.md` | 全局上下文、决定是否并行、汇总结果、统一对外 | 始终激活 |
| **Requirement Analyst** | `requirement-analyst.md` | 目标、范围、成功标准、边界 | 始终激活 |
| **Impact Analyst** | `impact-analyst.md` | 产品/领域影响扫描 | 始终激活 |
| **Challenger** | `challenger.md` | 边界、反例、误用、极端场景 | 中/高风险或高模糊度 |
| **Project Surface Router** | `project-surface-router.md` | 承载面缩圈 | 始终激活 |
| **Deep Dive Specialist** | `deep-dive-specialist.md` | 高风险制品定点深挖 | 仅高风险点 |
| **Clarification Synthesizer** | Lead 内建 | 汇总归类决策 | 始终激活 |

**并行策略**：

| 复杂度 | 并行度 | 配置 |
|--------|--------|------|
| 简单需求 | 1 主串行 | Lead 串行推进 |
| 中等需求 | 1 主 + 2~3 辅 | Lead + Impact Analyst + Surface Router |
| 复杂需求 | 1 主 + 3~5 辅 | Lead + 全部角色 agent teams 并行 |

**对外输出规则**：
- 只允许 Lead agent 统一收口
- 不允许多个 agent 分别向用户提问
- 所有 must_confirm 必须压缩进一个统一 Decision Packet

### 7.4 EXECUTE — 开发执行

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 获取可执行 task → 2. runtime preflight → 3. worker 在 re-anchor 后实现 → 4. task smoke + evidence 校验 → 5. 原子提交 + receipt → 6. 循环 |
| **主负责组件** | deep-implement |
| **内循环** | re-anchor → preflight → 实现 → 验证 → task smoke → evidence 校验 → triage/提交 → 下一个 |

**V4.3 运行时 Harness 要求**：

| 检查点 | 职责 | 失败处理 |
|--------|------|---------|
| **Preflight** | 校验输入上下文、依赖、工作区、预算 | 阻断当前 task |
| **In-loop Eval** | 漂移检测、失败计数、证据完整性 | 回流 PLAN 或升级人工 |
| **Task Smoke** | 每个 task 完成后最小可运行验证 | 标记失败，不提交 |
| **Stage Smoke** | EXECUTE 出口做阶段级烟测 | 阻断进入 VERIFY |
| **Auto-Diagnose** | 失败时收集日志、diff、测试结果 | 输出诊断报告 |

**新增发现问题处理规则**：
- `local_fix`：当前 task 内可修复
- `plan_patch`：需要修改 plan，回流 PLAN
- `spec_patch`：需要修改 spec，回流 SPEC

### 7.5 VERIFY — 验证与审查（V4.3 增强版）

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 汇总 runtime receipts → 2. 跨模型技术 review → 3. spec compliance 审查 → 4. 安全审查 → 5. 对抗式补盲 → 6. 验收议会 → 7. stage smoke |

#### 7.5.1 对抗式补盲（V4.3 通用版）

不用偏系统术语，而用所有项目都能套用的问题维度：

| 问题维度 | 检查内容 | 输出分类 |
|---------|---------|---------|
| 功能行为 | 核心功能是否按 spec 实现 | 已测并通过 / 未测但可接受 / 未测且必须补 |
| 状态变化 | 状态转换是否覆盖所有路径 | 同上 |
| 正确性 | 不变量是否被维护 | 同上 |
| 性能资源 | 是否在预算内 | 同上 |
| 兼容演进 | 旧行为是否被破坏 | 同上 |
| 安全权限 | 是否有未授权访问路径 | 同上 |
| 可观测性 | 关键路径是否有日志/指标 | 同上 |
| 用户体验 | 误用场景是否有防护 | 同上 |

### 7.6 FIX — 问题修复

- review → fix 循环，最多 3 轮，超出升级人工
- 每轮 fix 必须明确对应的 finding
- 修复后必须重跑受影响 smoke / 测试
- 不允许"顺手额外改一批"而不更新 evidence
- 受 runtime harness 约束

## 八、分层议会详细设计

| 议会类型 | 阶段 | 规模 | 角色 | 裁决 |
|---------|------|------|------|------|
| **轻议会** | SPEC 出口 | 3~5 | 规格完整性、一致性、可落地性、可测试性、安全约束 | `GO` / `REVISE` / `HOLD` |
| **计划议会** | PLAN 出口 | 5~7 | 覆盖性、任务切分、依赖、测试策略、架构、安全、恢复 | `READY` / `READY_WITH_CONDITIONS` / `BLOCK` |
| **验收议会** | VERIFY 出口 | 5~7 | 代码质量、业务逻辑、安全、测试、影响范围、边界 | `PASS` / `FAIL` |
| **发布议会** | DONE 出口 | 3~4 | Release readiness、Security signoff、Delivery、Learning governance | `RELEASE_READY` / `RELEASE_WITH_CONDITIONS` / `NOT_READY` |

**议会实现**：
- 轻议会 / 计划议会 / 发布议会：harness 内建 reviewer agents 并行调度
- 验收议会：复用 Ring `default` 的 `/ring:codereview` 7 路并行审查

**自动放行规则**：
- Decision Bundle 全部为 assumable/deferrable → 自动放行
- Decision Bundle 有 must_confirm → 阻断
- 议会 verdict = GO/PASS/READY/RELEASE_READY → 自动放行
- 议会 verdict = REVISE/HOLD/BLOCK → 阻断

`/harness:auto {epic-id}` 启动全流程自治循环：

```text
project-profile → clarify → spec → [轻议会] → plan → [计划议会] →
work(task1) → work(task2) → ... → review → [验收议会] → done → [发布议会]
```

**风险分级规则**：
- 低风险 feature → 可全程自治
- 中风险 feature → 在关键节点暂停等人确认
- 高风险 feature → 不启用完全自治

**自治停止条件**：
- 触发 must_confirm 且预算允许中断
- 安全 / 合规 reviewer FAIL
- 同一 task 连续 3 次失败
- stage smoke 失败
- drift detector 判定偏离 spec

### 14.1 三层沉淀模型

| 层 | 存储 | 内容 |
|---|------|------|

| 团队级 | CLAUDE.md / rules | 通用工程约定 |

```text
运行记录 / 决策 / receipts / triage / 验证结果
    → 模式挖掘（skill-miner agent）
    → candidate-skill.md
    → shadow validate
    → auto-refine
    → 人工晋升
```

### 14.3 安全边界

- 只允许自动更新项目级候选 Skill
- 不允许未经人工审核直接写入团队级规则
- 低置信度模式只允许保留为 observation
- shadow 期间出现误导 → 立即降级或废弃

## 十五、失败路径设计

| 场景 | 处理 |
|------|------|
| 需求不可行 | CLARIFY 输出 `not-feasible` + 原因，终止 |
| 技术方案不成立 | 计划议会 BLOCK，回流 SPEC 或终止 |
| 外部依赖不可用 | task 标记 `blocked` |
| Epic 被取消 | 标记关闭，保留产物 |
| Token 预算耗尽 | 暂停，保存进度 |
| 自治循环异常 | guard hooks 暂停，输出错误报告 |
| 连续 task 失败 3 次 | 自动 triage 后回流 PLAN 或升级人工 |
| stage smoke 失败 | 阻断后进入 FIX |
| 候选 Skill shadow 失败 | 降级为 observation |
| 中断预算耗尽但仍需人拍板 | 仅安全/合规/业务方向允许破例，写入审计 |

## 十九、关键风险与缓解

| 风险 | 描述 | 缓解 |
|------|------|------|
| **CLARIFY agent 体系过重** | 7 个角色 agent 全部激活成本极高 | 默认 1 主串行；复杂场景再启用并行；动态强度控制 |
| **影响扫描流于空泛** | 不深入代码的扫描可能产出空洞结论 | 固定输出格式（影响面清单 + 高优先级问题 + 后续分析指引）；结合项目画像聚焦 |
| **Unknowns Ledger 维护成本** | 多阶段手动更新台账工作量大 | 自动化更新脚本；每阶段出口自动校验 |
| **Coverage Matrix 流于形式** | 列表做了但映射不实际 | PLAN 阶段强制校验"问题→task→验证→证据"；计划议会重点审查 |
| **中断预算过严导致误推进** | AI 为了不打断人而错过关键问题 | 安全/合规/业务方向保留破例通道；Decision Packet 必须附带"不回答时默认怎么做" |
| **ShipSpec → deep-plan 格式断层** | 两者产物格式不兼容 | 优先实现桥接脚本；备选直接用 SDD.md |
| **组件自动触发导致职责越界** | 多插件并存时 hook 冲突 | 严格裁剪规则；ShipSpec 不安装 Stop 钩子；Ring 仅 default |
| **自治模式安全风险** | 全自动运行可能产生不可逆变更 | guard hooks + 中断预算 + 人工检查点 + 高风险禁用完全自治 |
| **Token 与延迟失控** | 多阶段多 reviewer 成本高 | 动态强度控制；Token 预算上限；低风险路径简化 |
| **自动进化 Skill 产生错误模式** | 规则膨胀、冲突 | 只自动更新项目级候选；团队级必须人工审核；shadow 失败立即降级 |

## 二十四、总结

V4.3 开发方案的核心变化：

1. **CLARIFY 阶段全面重建**：从"Superpowers brainstorming"升级为"Lead + 多角色 agent 协作体系"，新增产品/领域影响扫描、决策分类、中断预算、Decision Packet 硬约束。

2. **两个贯穿全程的新产物**：`unknowns-ledger.json`（问题台账）和 `coverage-matrix.json`（覆盖映射矩阵），把问题发现→规格固化→实施承接→验证闭环→经验沉淀串成一条链。

3. **运行时 Harness 引擎**：EXECUTE 阶段从"调用 deep-implement"升级为"preflight → in-loop eval → task smoke → stage smoke → auto-diagnose → rollback/re-anchor/re-plan"的全程控偏体系。
