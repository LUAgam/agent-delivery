> 在 V4.1 定向修订版基础上，继续补入两项正式机制：
>
> 1. 多代码仓库场景下的加速且准确的分析策略
> 2. 需求澄清阶段“先脱离仓库，再结合仓库，再定点深挖”的分阶段策略

> 其余结论与保留项沿用 V4.1，不额外扩展未指定缺口。

## 二、最终主张

> **stage-harness** 是一个自包含的 Claude Code 插件，在接收到模糊需求后，自动完成需求澄清、规格定义、计划生成、开发执行、验证审查、交付发布的全流程推进。整个过程中只把极少数关键决策点交给人，其余全部自主完成。

核心特征：

1. **一个插件满足核心主链诉求**——安装 stage-harness 即获得完整能力
2. **接住模糊输入**——brainstorming + interview 自动澄清
3. **AI 自主判断**——决策分类机制区分“必须问人 / 可先假设 / 可延后”
4. **端到端交付**——从需求到代码 + 测试 + 交付包
5. **多角色议会**——4 类分层议会做多视角质量保障
6. **自治运行**——Ralph 模式支持 overnight 无人推进
7. **运行时 harness**——执行中持续自检、评估、烟测、回退、回流
8. **技能进化**——从运行记录中挖掘可复用模式，自动形成候选 Skill 并迭代
9. **最少打断**——中断预算 + 问题批处理 + 硬门禁，避免零碎打扰
10. **多仓分析**——统一工作区 + 跨仓索引 + repo-router + 子代理下钻
11. **分阶段澄清**——先做业务澄清，再做结合仓库的落地澄清，最后只对高风险区域深挖

## 三、不可替换的机制层

以下机制是整个体系的骨架，不随实现方式变化。

### 3.1 PLAN 是后半程控制中心

后半程主链：`PLAN → EXECUTE → VERIFY → FIX → DONE`

硬规则：
- EXECUTE 发现切分失效 → 必须回流 PLAN
- EXECUTE 发现依赖不成立 → 必须回流 PLAN
- 运行时 harness 判定任务漂移或反复失败 → 必须回流 PLAN
- 不允许“计划之外的临场扩写”静默混入

### 3.2 Decision Bundle + 决策分类

在关键阶段出口收束决策。AI 主动判断哪些可先假设推进，只把真正无法自主决策的问题交给人。

| 分类 | 处理方式 | 示例 |
|------|---------|------|
| **must_confirm** | 阻断，等人拍板 | 目标用户是内部还是外部？ |
| **assumable** | AI 假设推进，记录依据，**不阻断** | 数据库默认用 PostgreSQL |
| **deferrable** | 不阻塞当前阶段，标记后续确认 | 日志格式的具体字段定义 |

自动放行规则：
- Decision Bundle 全部为 assumable / deferrable → **自动放行**
- Decision Bundle 有 must_confirm → 阻断
- 议会 verdict = GO / PASS / READY / RELEASE_READY → **自动放行**
- 议会 verdict = REVISE / HOLD / BLOCK → 阻断

### 3.3 中断预算与问题批处理（新增，硬约束）

“尽量少打断人”不再只是原则，而是**运行时必须遵守的硬约束**。

#### 中断预算

每个 epic 在 CLARIFY 完成后即写入 `interrupt-budget.json`：

| 风险等级 | 强制中断预算 | 默认允许中断点 |
|---------|-------------|----------------|
| 低风险 | **1 次** | VERIFY→DONE |
| 中风险 | **2 次** | PLAN→EXECUTE、VERIFY→DONE |
| 高风险 | **3 次** | CLARIFY、PLAN→EXECUTE、VERIFY→DONE |

规则：
- 预算只统计 **must_confirm 阻断式中断**
- assumable / deferrable 不计入中断预算
- 超出预算后，除非触发安全/合规/业务方向/不可逆选型，否则必须转为 assumable 或 deferrable
- 同阶段禁止多次零碎追问；必须合并成一个 **Decision Packet**

#### 问题批处理规则

- **同一阶段只允许一个未决 Decision Packet**
- 同一个 Packet 内问题必须按优先级排序：`必须现在回答 > 可给默认值 > 可延后`
- AI 必须先尝试：检索上下文 → 套用既有规则 → 生成默认假设 → 评估是否可延后
- 只有仍无法安全推进时才允许发起 Packet
- Packet 必须附带：`为什么现在必须问`、`不回答时默认怎么做`、`最晚何时需要答案`

Decision Packet 结构：

```json
{
  "packet_id": "DP-PLAN-01",
  "stage": "PLAN",
  "interrupt_budget_total": 2,
  "interrupt_budget_used": 1,
  "questions": [
    {
      "id": "D-PLAN-03",
      "question": "主后端框架是否允许从 Flask 假设为 FastAPI？",
      "classification": "must_confirm",
      "why_now": "影响 task 切分与目录结构，后续返工成本高",
      "default_if_no_answer": null,
      "needed_by": "EXECUTE"
    },
    {
      "id": "D-PLAN-04",
      "question": "日志字段命名是否采用 snake_case？",
      "classification": "deferrable",
      "why_now": "不阻塞主线实现",
      "default_if_no_answer": "按团队既有 convention",
      "needed_by": "DONE"
    }
  ]
}
```

### 3.4 分层议会（不可降级的核心机制）

| 议会类型 | 阶段 | 规模 | 关注重点 |
|---------|------|------|---------|
| 轻议会 | SPEC | 3~5 | 漏项、冲突、隐含假设 |
| 计划议会 | PLAN | 5~7 | 覆盖、切分、依赖、风险 |
| 验收议会 | VERIFY | 5~7 | 代码质量、安全、测试 |
| 发布议会 | DONE | 3~4 | 交付完整性、安全签署 |

### 3.5 阶段门禁 + 可审查 / 可阻断 / 可回退

每个阶段出口有结构化检查，不合格则阻断，出问题可回退。

### 3.6 运行时 Harness（新增，不可降级）

stage-harness 不只在阶段出口做检查，**还要在执行过程中持续控偏**。

运行时 harness 的硬职责：
- **Preflight**：任务开跑前校验输入上下文、依赖、工作区、预算
- **In-loop Eval**：任务执行中做漂移检测、失败计数、证据完整性检查
- **Task Smoke**：每个 task 完成后做最小可运行验证
- **Stage Smoke**：每个阶段出口做阶段级烟测
- **Auto-Diagnose**：失败时自动收集命令、日志、diff、测试结果、环境摘要
- **Rollback / Re-anchor / Re-plan**：按失败类型自动采取纠偏动作

执行中不满足以下任一条件，必须阻断并处理：
- 任务目标与 spec 不一致
- 依赖前置条件未满足
- 代码通过实现但未留下可验证 evidence
- 连续失败超过阈值
- 回归烟测失败


### 3.7 多仓分析策略（新增，不可降级）

项目包含多个代码仓库时，禁止让主 Agent 直接把所有仓库当成一个大目录硬扫。  
默认采用以下四层组合：

1. **统一工作区**：把相关 repo 纳入同一工作区，但保留仓库边界
2. **跨仓索引**：优先依赖代码搜索 / 符号导航 / 引用关系，而不是全文硬读
3. **repo-router**：先判断问题涉及哪些仓，再决定由哪些子代理下钻
4. **证据式读取**：先定位符号、调用链、接口和配置，再读取正文源码

硬规则：
- 禁止在早期阶段对全部仓库做无差别全量扫描
- 先做 `workspace-manifest.yaml`，记录 repo 角色、语言、入口、owner、上下游依赖
- 先做 repo-router，再做仓内阅读
- 优先使用“关键词路由 → 符号导航 → 定点源码阅读”三段式
- 主代理只做跨仓汇总；仓内细读优先交给子代理

推荐分析链路：

`问题输入 → repo-router → 命中仓库集合 → 符号/引用/依赖定位 → 子代理分仓下钻 → 主代理汇总`

### 3.8 澄清与仓库分析的分阶段策略（新增，不可降级）

需求澄清默认采用三段式，而不是一上来扎进代码：

#### 第一段：脱离仓库的业务澄清
先澄清：
- 目标是什么
- 成功标准是什么
- 用户是谁
- 边界在哪里
- 哪些点必须让人拍板

这一段的目标是**定义“要做什么”**，避免被现有实现绑架。

#### 第二段：结合仓库的落地澄清
在高层目标、范围、成功标准初步明确后，再引入代码仓库，回答：
- 需求落在哪几个 repo / 模块
- 已有能力覆盖到什么程度
- 哪些可以复用，哪些必须新增
- 当前架构与历史包袱的约束是什么

这一段的目标是**确认“在现有系统里怎么落”**。

#### 第三段：定点深挖
只有命中高风险模块、跨仓链路、关键接口、核心数据模型时，才允许深入读代码、追调用链、查配置和历史实现。

硬规则：
- 默认先做业务澄清，再做落地澄清
- 只有在需求本质是排障 / 兼容 / 迁移 / 重构 / 已有系统改造时，才允许提前引入仓库分析
- 禁止在 CLARIFY 早期就全量扫描所有仓库
- 所有仓库分析都必须先经过 repo-router

## 五、参考复用策略

### 5.1 从 Flow-Next 内化什么

Flow-Next（v0.26.1）是一个高度成熟的系统。stage-harness 参考其设计和实现，内化以下能力：

| 能力 | Flow-Next 原实现 | stage-harness 内化方式 |
|------|-----------------|---------------------|
| **epic/task 模型** | `.flow/` 目录 + flowctl CLI | 复用 epic/task JSON 格式 + 自建或 fork flowctl |
| **plan 生成** | `/flow-next:plan`（并行 scouts 调研） | 内化为 `/harness:plan` skill，保留 scouts 并行机制 |
| **任务执行** | `/flow-next:work`（worker subagent + re-anchor） | 内化为 `/harness:work` skill，保留 worker 隔离 + re-anchor |
| **跨模型 review** | impl-review / plan-review（RepoPrompt/Codex/Export） | 内化为 `/harness:review` skill |
| **interview** | `/flow-next:interview`（深度 Q&A） | 内化为 `/harness:interview` skill |
| **依赖管理** | `flowctl dep` / `ready` / `next` | 直接复用 flowctl 或内建等价 CLI |
| **自治循环** | Ralph（hooks + guard + receipts） | 内化为 `/harness:auto` skill，保留 guard + receipts |
| **memory** | `flowctl memory`（pitfalls/conventions/decisions） | 直接复用或内建等价能力 |
| **worktree** | `worktree.sh`（多分支隔离） | 复用脚本 |

### 5.2 从 Superpowers 内化什么

| 能力 | Superpowers 原实现 | stage-harness 内化方式 |
|------|-------------------|---------------------|
| **brainstorming** | brainstorming skill（发散式探索） | 内化为 `/harness:clarify` skill 的前半段 |
| **TDD 方法纪律** | TDD + review 方法约束 | 内化为 EXECUTE 阶段的 skill 规则 |
| **执行纪律** | plan discipline | 内化为 PLAN/EXECUTE 的 skill 规则 |

### 5.3 从 ECC 内化什么

| 能力 | ECC 原实现 | stage-harness 内化方式 |
|------|----------|---------------------|
| **安全审查** | security-reviewer agent | 内化为验收议会的安全 reviewer 角色 |
| **audit** | 审计记录能力 | 内化为 `.harness/` 的结构化审计日志 |
| **safety gate** | 安全门禁 | 内化为阶段门禁的安全检查项 |

## 七、命令体系

| 命令 | 功能 | 阶段 |
|------|------|------|
| `/harness:start {模糊需求}` | 从模糊需求启动全流程，自动走 CLARIFY → SPEC → PLAN → EXECUTE → VERIFY → DONE | 全流程 |
| `/harness:clarify {模糊需求}` | 需求澄清（brainstorming + interview + 决策分类 + 中断压缩） | CLARIFY |
| `/harness:spec {epic-id}` | 规格定义（epic 创建 + 精炼 + 轻议会） | SPEC |
| `/harness:plan {epic-id}` | 计划生成（scouts 并行调研 + task 图谱 + 计划议会） | PLAN |
| `/harness:work {epic-id 或 task-id}` | 开发执行（worker + re-anchor + runtime harness + 原子提交） | EXECUTE |
| `/harness:review {epic-id}` | 审查（跨模型技术 review + 验收议会） | VERIFY |
| `/harness:auto {epic-id}` | 自治模式（自动循环 plan → work → review → done） | 全流程自治 |
| `/harness:status` | 查看当前 epic/task 状态、阶段进度、Decision Bundle 状态、中断预算、运行时健康度、当前命中仓库 | 任意 |
| `/harness:done {epic-id}` | 交付（发布议会 + 交付包 + 经验沉淀 + 候选 Skill） | DONE |

**`/harness:start` 是核心入口**——用户丢进一段模糊需求，插件自动走完全流程。等同于依次调用 clarify → spec → plan → work → review → done，中间只在 must_confirm 的 Decision Packet 处暂停等人。

### 8.3 Session 恢复与 Handoff

长流程必然跨越多个 Claude Code session（token 耗尽、网络中断、手动退出）。stage-harness 必须支持断点续做：

**自动恢复**：`stage-init.sh`（SessionStart hook）检测 `.harness/features/` 是否有进行中的 epic，如果有则：
1. 读取 `handoff.md` 恢复上下文
2. 提示用户当前进度（“Epic fn-3 处于 EXECUTE 阶段，已完成 4/7 个 task”）
3. 展示当前中断预算与运行时健康度
4. 询问是否继续

**自动保存**：每个阶段完成时自动写入 `handoff.md`：

```markdown
## 当前状态
- Epic: fn-3-user-auth
- 阶段: EXECUTE
- 进度: 4/7 tasks 完成
- 中断预算: 1/2 已使用
- 运行时健康度: GREEN

## 下一步
- 执行 task fn-3.5（用户权限校验）

## 关键上下文
- 使用 JWT + RBAC 方案（Decision D-CLARIFY-02 已确认）
- 数据库用 PostgreSQL（假设推进，未被推翻）
- 最近一次 smoke 通过；最近一次 triage 无新增 blocker
```

### 8.4 上下文窗口管理

全流程产物量大，不可能全部加载到 LLM 上下文中。每个阶段只加载当前相关 artifacts：

| 阶段 | 加载 | 不加载 |
|------|------|--------|
| CLARIFY | 原始需求文本 + 中断预算规则 | 无 |
| SPEC | clarification-notes + decision-bundle | 上游对话 |
| PLAN | epic spec（specs/*.md）+ 风险等级 + 预算状态 | 原始需求、CLARIFY 细节 |
| EXECUTE | 当前 task spec + epic spec 摘要 + 上一轮运行回执 | 其他 task、议会报告 |
| VERIFY | epic spec + 全量 diff + task evidence + smoke 汇总 | 执行过程细节 |
| DONE | verification.json + delivery-summary 模板 + skill candidates | 执行过程 |

**原则**：每个阶段的输入是上一阶段的 artifacts，不是完整对话上下文。re-anchor 机制从 EXECUTE 扩展到所有阶段——每进入新阶段都重新锚定相关上下文。


### 8.5 多仓上下文加载规则（新增）

多仓场景下，上下文加载必须遵守“**先路由、再缩圈、后深读**”：

| 步骤 | 默认加载 | 禁止默认加载 |
|------|---------|-------------|
| repo-router | `workspace-manifest.yaml`、目录级信息、入口文件、owner、依赖摘要 | 全仓正文源码 |
| 仓库初筛 | 命中 repo 的 README、接口定义、核心目录树、关键配置 | 其余未命中仓库源码 |
| 定点深挖 | 命中符号的定义/引用/实现、相关测试、配置、调用链 | 无关模块与无关历史文件 |

硬规则：
- 单轮默认深读仓库数不超过 3 个，除非计划议会批准扩大范围
- 单个子代理一次只负责 1 个仓库或 1 条跨仓链路
- 主代理的汇总结论必须标注：涉及 repo、关键文件、调用链、证据位置、置信度

## 九、各阶段设计

### 9.0 需求入口标准化

模糊需求直接作为 `/harness:start` 或 `/harness:clarify` 的输入。插件内部自动做格式化：

> 将输入整理为：1. 一句话目标；2. 已知约束；3. 期望交付物。信息不足标注“待澄清”。

### 9.1 CLARIFY — 需求澄清

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 业务澄清（不读仓库）→ 2. brainstorming（发散探索）→ 3. interview（收敛精炼）→ 4. 落地澄清（repo-router + 命中仓初筛）→ 5. 必要时定点深挖 → 6. 决策分类 → 7. 中断压缩 → 8. Decision Bundle / Packet |
| **产物** | `clarification-notes.md`、`decision-bundle.json`、`decision-packet.json`、`interrupt-budget.json`、`repo-routing.json`（如已引入仓库） |
| **出口** | must_confirm 项已按 Packet 统一发问；assumable 项已记录假设依据；若引入仓库，则已明确命中 repo 与落地约束 |

#### CLARIFY 阶段的仓库介入规则

默认规则：
- **先脱离仓库做业务澄清**
- **再结合仓库做落地澄清**
- **最后只对关键区域定点深挖**

允许提前引入仓库分析的场景：
- 已有系统改造
- 排障 / 兼容 / 迁移 / 重构
- 明显依赖历史实现，脱离仓库无法判断成本和风险
- 用户给出的目标已经很明确，但落地边界高度依赖现状代码

即使提前引入仓库，也必须遵守：
- 先跑 repo-router
- 先看入口、接口、配置、README、目录树
- 不得在 CLARIFY 早期全量扫全部 repo
- 深读源码只针对已命中的高风险点

**决策分类判断规则**：

| 条件 | 分类 | 依据 |
|------|------|------|
| 涉及业务方向选择（to B / to C / 定价 / 目标客群） | must_confirm | 业务决策不能由 AI 代拍 |
| 涉及合规 / 法律 / 安全红线 | must_confirm | 风险不可逆 |
| 涉及不可逆的技术选型（如主语言 / 主框架） | must_confirm | 变更成本极高 |
| 有行业惯例 + 项目上下文支撑 + 后续可纠正 | assumable | 可假设推进 |
| 属于实现细节 + 不影响架构 + 后续可改 | deferrable | 不阻塞主线 |
| AI 对分类没有信心时 | must_confirm | 保守处理 |

Decision Bundle 结构：

```json
{
  "id": "D-CLARIFY-01",
  "question": "数据库选型",
  "classification": "assumable",
  "assumption": "默认 PostgreSQL",
  "assumption_basis": "团队现有技术栈 + 无特殊性能要求",
  "override_needed_by": "PLAN",
  "human_decision": null
}
```

**CLARIFY 阶段的中断压缩规则**：
- 先尽最大可能产出 `assumable` 与 `deferrable`
- 再将剩余 `must_confirm` 压缩进一个 Packet
- 不允许边问边补、连续追问、同义问题拆开发
- 如果问题数量超过 5 个，必须再次做优先级压缩，只保留当前阶段真正阻断的事项

**典型场景强制中断上限**：

| 风险等级 | 强制中断上限 | 说明 |
|---------|-------------|------|
| 低风险 | **最多 1 次** | 一般不在 CLARIFY 打断 |
| 中风险 | **最多 2 次** | CLARIFY 仅在存在不可替代业务决策时打断 |
| 高风险 | **最多 3 次** | 可在 CLARIFY 产生 1 次高价值中断 |

### 9.2 SPEC — 规格定义

| 项目 | 内容 |
|------|------|
| **工作流** | 1. epic 创建 → 2. 写入规格（epic spec） → 3. interview 精炼 → 4. 轻议会审查 → 5. Decision Bundle |
| **产物** | `.harness/specs/{epic-id}.md`、`decision-bundle.json` |
| **出口** | 轻议会 verdict = GO + Decision Bundle 已处理 |

Epic spec 模板：

```markdown
# {Epic Title}

## Goal & Context
## Architecture & Data Models
## API Contracts
## Edge Cases & Constraints
## Acceptance Criteria
## Boundaries
## Decision Context
## Test Strategy
## Risk Register
```

### 9.3 PLAN — 实施计划

| 项目 | 内容 |
|------|------|
| **工作流** | 1. repo-router 缩圈 → 2. 并行 scouts 调研（repo-scout + docs-scout + practice-scout + context-scout + dependency-mapper）→ 3. 生成 task 图谱（依赖排序）→ 4. 技术 plan-review → 5. 计划议会 → 6. Decision Bundle / Packet |
| **产物** | `.harness/tasks/*.json + *.md`（task 图谱）、`plan-council-verdict.json`、`decision-packet.json`、`repo-routing.json`、`dependency-graph.json` |
| **出口** | plan-review 通过 + 计划议会 verdict ≠ BLOCK + Decision Bundle 已处理 |

**PLAN 阶段的多仓规则**：
- repo-router 必须先给出“主命中仓 / 次命中仓 / 观察仓”
- 优先按“入口仓 → 依赖仓 → 基础库仓”顺序分析
- 任务拆分必须标注归属 repo
- 跨仓 task 必须显式写明接口边界、依赖方向、联调/回归点
- 若需要扩大到 4 个以上深读仓库，需由计划议会确认

**PLAN 阶段的中断规则**：
- 只能在 task 图谱已成型后统一发起 Packet
- 若预算剩余为 0，则除安全/业务/不可逆选型外禁止中断，必须改为默认假设推进
- Packet 需携带“若默认推进，未来在哪个节点还能补救”

### 9.4 EXECUTE — 开发执行

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 获取可执行 task（依赖排序）→ 2. runtime preflight → 3. worker subagent 在 re-anchor 后实现 → 4. task smoke + evidence 校验 → 5. 原子提交 + receipt → 6. 循环直到全部完成 |
| **产物** | 代码、测试、evidence（commits / tests / PRs）、runtime receipts、git 原子提交 |
| **内循环** | re-anchor → preflight → 实现 → 测试 → task smoke → evidence 校验 → triage/提交 → 下一个 task |

#### EXECUTE 内的运行时 harness

每个 task 强制执行以下步骤：

1. **Preflight**
   - 读取当前 task spec、依赖状态、上轮失败摘要
   - 确认工作区干净、目标文件范围明确、预算未超限
   - 生成本轮执行计划（不超过 5 步）

2. **Implement**
   - 仅在 task 边界内改动
   - 多仓任务必须标注当前改动发生在哪个 repo
   - 发现新依赖或需求漂移时立即标记，不得静默扩写

3. **Verify in-loop**
   - 至少执行与 task 直接相关的本地测试/检查
   - 无测试时，必须执行最小可运行 smoke
   - 保留命令、结果、退出码

4. **Task Smoke**
   - 功能级最小验证，而非只看代码 diff
   - smoke 失败时进入 triage，不允许直接提交

5. **Evidence & Receipt**
   - 写入 `runtime-receipt.json`
   - 记录：目标、改动文件、执行命令、结果摘要、失败原因、下一步建议

6. **Failure Triage**
   - 第 1 次失败：自动重试或 re-anchor
   - 连续 2 次失败：自动收集诊断包
   - 连续 3 次失败：回流 PLAN 或升级人工

runtime receipt 示例：

```json
{
  "task_id": "fn-3.5",
  "status": "passed",
  "preflight": "ok",
  "commands": ["npm test -- user-permission", "npm run lint"],
  "smoke": "passed",
  "evidence": ["commit abc123", "test report saved"],
  "drift_detected": false,
  "next_action": "task fn-3.6"
}
```

### 9.5 VERIFY — 验证与审查

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 汇总 runtime receipts → 2. 跨模型技术 review → 3. spec compliance 审查 → 4. 安全审查 → 5. 验收议会 → 6. stage smoke |
| **产物** | review 结果、`verification.json`、安全报告、smoke 汇总 |
| **出口** | 技术 review SHIP + 验收议会 PASS + stage smoke 通过 |

三层 review：

| 层 | 关注 | 方式 |
|----|------|------|
| 技术 review | 代码质量、逻辑正确性 | 跨模型（Claude 实现 → GPT 审查） |
| Spec compliance | 实现与规格的一致性 | 自动 |
| 治理 review | 安全、团队标准、影响范围 | 多角色验收议会（5~7 reviewer） |

**运行时 harness 在 VERIFY 的附加职责**：
- 校验所有 task 是否都有 receipt
- 校验关键 task 是否有至少 1 个可运行 smoke 证据
- 自动识别“代码通过但证据不足”的伪完成状态
- 如发现伪完成，直接回流 FIX

### 9.6 FIX — 问题修复

review → fix 循环。最多 3 轮，超出升级人工。

FIX 阶段同样受 runtime harness 约束：
- 每轮 fix 必须明确对应的 review finding
- 修复后必须重跑受影响 smoke / 测试
- 不允许“顺手额外改一批”而不更新 evidence

### 9.7 DONE — 交付与沉淀

| 项目 | 内容 |
|------|------|
| **工作流** | 1. 发布议会 → 2. 生成交付包 → 3. 经验提名 → 4. 候选 Skill 挖掘 → 5. shadow 验证 → 6. 受控晋升 |
| **产物** | `release-notes.md`、`delivery-summary.md`、`learning-candidates.md`、`candidate-skills/` |
| **出口** | 发布议会 verdict = RELEASE_READY |

DONE 阶段新增：
- 从 `runs/`、`observations/`、`decisions/`、`receipts/` 中挖掘稳定模式
- 自动生成 project-local 候选 Skill
- 候选 Skill 先进入 **shadow 模式**，不直接变成团队级规则
- 只有通过 shadow 验证且人工审核通过，才允许晋升到团队级

## 十、分层议会详细设计

分层议会是**不可降级的核心机制**。

### A 类：轻议会（SPEC 出口）

| 项目 | 内容 |
|------|------|
| **规模** | 3~5 reviewer subagent |
| **角色** | 规格完整性、一致性、可落地性、可测试性（可选）、安全约束（按需） |
| **裁决** | `GO` / `REVISE` / `HOLD` |
| **输出** | `spec-council-report.md`、`spec-council-verdict.json` |

### B 类：计划议会（PLAN 出口）

| 项目 | 内容 |
|------|------|
| **规模** | 5~7 reviewer subagent |
| **角色** | 覆盖性、任务切分、依赖路径、测试策略、架构承接、安全风险、回滚（按需） |
| **核心问题** | 需求全覆盖？task 边界清晰？依赖无循环？高风险有缓解？冷启动可执行？ |
| **裁决** | `READY` / `READY_WITH_CONDITIONS` / `BLOCK` |
| **输出** | `plan-council-report.md`、`plan-council-verdict.json` |

### C 类：验收议会（VERIFY 出口）

| 项目 | 内容 |
|------|------|
| **规模** | 5~7 reviewer subagent |
| **角色** | 代码质量、业务逻辑、安全、测试质量、nil/null safety、影响范围、死代码 |
| **裁决** | `PASS` / `FAIL` |
| **输出** | `review-report.md`、`test-report.md`、`verification.json` |

### D 类：发布议会（DONE 出口）

| 项目 | 内容 |
|------|------|
| **规模** | 3~4 reviewer subagent |
| **角色** | Release readiness、Security signoff、Delivery/docs、Learning governance |
| **裁决** | `RELEASE_READY` / `RELEASE_WITH_CONDITIONS` / `NOT_READY` |
| **输出** | `release-council-report.md`、`release-council-verdict.json` |

### 裁决汇总逻辑

| 策略 | 适用 | 规则 |
|------|------|------|
| 多数决 | 轻议会、发布议会 | >50% 同意则放行 |
| 一票否决 | 安全相关 | 安全 reviewer FAIL 则整体 FAIL |
| 加权多数 | 计划议会、验收议会 | 覆盖性/安全权重 2x，其他 1x |

## 十一、动态强度控制

根据风险等级调整流程重量——**议会始终存在**，只调规模：

| 控制维度 | 低风险 | 中风险 | 高风险 |
|---------|-------|-------|-------|
| CLARIFY | 可跳过 brainstorming | brainstorming + interview | 完整 brainstorming + 深度 interview |
| 轻议会 | 3 reviewer | 3 reviewer | 5 reviewer |
| Plan depth | short | standard | deep |
| 计划议会 | 3 reviewer | 5 reviewer | 7 reviewer |
| EXECUTE 模式 | 自治（auto）+ 轻量 harness | worker + review + 标准 harness | worker + review + 强化 harness + 人工检查点 |
| 验收议会 | 3 reviewer | 5 reviewer | 7 reviewer |
| 发布议会 | 2 reviewer | 3 reviewer | 4 reviewer |
| Decision Bundle | 仅 PLAN→EXECUTE | 全部触发点 | 全部触发点 + 额外检查点 |
| 中断预算 | 1 次 | 2 次 | 3 次 |

## 十二、自治模式

`/harness:auto {epic-id}` 启动全流程自治循环：

```
clarify → spec → plan → [议会] → work(task1) → work(task2) → ... → review → [议会] → done
```

- 低风险 feature 可全程自治（hooks + guard + receipts）
- 中风险 feature 在关键节点（PLAN→EXECUTE、VERIFY→DONE）暂停等人确认
- 高风险 feature 不启用自治
- guard hooks 注入 token 预算检查和安全防护
- runtime harness 在自治时持续做 smoke、漂移检测、失败归因
- 所有运行记录保存在 `.harness/features/{epic-id}/runs/`

自治停止条件：
- 触发 must_confirm 且中断预算尚可使用
- 安全/合规 reviewer 报 FAIL
- 同一 task 连续 3 次失败
- stage smoke 失败
- drift detector 判定偏离 spec

## 十三、经验沉淀与技能进化

这部分从“经验沉淀”升级为“**经验沉淀 + 自动进化 Skill**”。

### 13.1 三层沉淀模型

| 层 | 存储 | 内容 | 生命周期 |
|----|------|------|---------|
| 项目级 | `.harness/memory/` | 当前项目的 pitfalls / conventions / runtime observations | 项目结束时归档 |
| 团队级 | CLAUDE.md / rules | 团队通用的工程约定 | 长期维护，定期清理 |
| 个人级 | 本地配置 | 个人偏好 | 个人维护 |

### 13.2 自动进化 Skill 的目标

系统不是只把经验记下来，而是要识别：
- 哪些澄清步骤反复出现且有效
- 哪些计划切分方式对同类任务成功率高
- 哪些故障排查路径可重复复用
- 哪些实现 + 验证组合能稳定通过

这些稳定模式应自动形成 **候选 Skill**，供后续项目复用。

### 13.3 候选 Skill 生成路径

路径：

`运行记录 / 决策 / receipts / triage / 通过结果 → 模式挖掘 → candidate-skill.md → shadow validate → auto-refine → 人工晋升（团队级）`

候选 Skill 模板：

```markdown
# Candidate Skill: auth-endpoint-task-smoke

## Trigger
当 task 涉及新增或修改鉴权接口时触发

## Preconditions
- 已有 API contract
- 已有最小测试入口

## Steps
1. 先跑权限相关现有测试
2. 修改实现
3. 跑目标测试
4. 做最小 smoke（未登录/已登录/越权）
5. 记录 receipt

## Boundaries
- 不处理跨服务权限联动
- 不替代完整集成测试

## Evidence
- 来自 4 次成功任务，成功率 100%

## Metrics
- use_count: 4
- success_rate: 1.0
- rollback_count: 0
```

### 13.4 Shadow 验证机制

候选 Skill 不直接正式启用，而是先进入 **shadow mode**：
- 系统在后续相似任务中“旁路观察”这个 Skill 是否仍然适用
- 记录命中率、成功率、回滚率、人工覆盖率
- 连续满足阈值后才升级为 project-local skill

建议阈值：
- `use_count >= 3`
- `success_rate >= 0.8`
- `rollback_count == 0` 或极低
- 无安全/合规副作用

### 13.5 自动迭代边界

为了避免“自学习污染”，自动进化必须受控：
- **只允许自动更新项目级 Skill 候选**
- **不允许未经人工审核直接写入团队级规则**
- 低置信度模式只允许保留为 observation，不得晋升
- 一旦 shadow 期间出现明显误导，立即降级或废弃

### 13.6 晋升路径

晋升路径：
- 项目经验 → `learning-candidates.md`
- 候选 Skill → `candidate-skills/`
- Shadow 验证通过 → `project-local skills`
- 人工审核通过 → `team rules / team skills`

### 13.7 清理策略

- 团队规则不超过 30 条，每季度 review
- project-local candidates 超过 20 个时按命中率和成功率淘汰尾部
- 连续 3 个项目未命中的 Skill 候选自动归档

## 十四、失败路径设计

| 场景 | 处理 |
|------|------|
| 需求不可行 | CLARIFY 输出 `not-feasible` + 原因，终止 |
| 技术方案不成立 | 计划议会 BLOCK，回流 SPEC 或终止 |
| 外部依赖不可用 | task 标记 `blocked`，等待解除 |
| Epic 被取消 | 标记关闭，保留产物 |
| Token 预算耗尽 | 暂停，保存进度，等恢复 |
| 自治循环异常 | guard hooks 暂停，输出错误报告 |
| 连续 task 失败 3 次 | 自动 triage 后回流 PLAN 或升级人工 |
| stage smoke 失败 | 阻断 DONE/继续执行，进入 FIX |
| 候选 Skill shadow 失败 | 降级为 observation，不再自动复用 |
| 中断预算耗尽但仍需人拍板 | 仅安全/合规/业务方向问题允许破例中断，并写入审计 |

## 十六、关键风险与缓解

| 风险 | 缓解 |
|------|------|
| **内化 Flow-Next 工作量大** | fork flowctl.py + 复用 skill/agent markdown，不从零开始 |
| **议会 token 成本高** | 动态强度控制（低风险 3 reviewer，高风险 7 reviewer） |
| **自治模式安全风险** | guard hooks + token 预算 + 关键节点人工检查点 |
| **单插件体量过大** | 按 skill 模块化，按需加载；命令可独立使用 |
| **与 Flow-Next 上游更新同步** | 记录 fork 基线版本，定期合并上游改进 |
| **memory 过早引入** | MVP-3 再加，MVP-1/2 用纯文件 |
| **运行时 harness 增加执行开销** | 低风险场景使用轻量 smoke，高风险场景启用强化 harness |
| **自动进化 Skill 产生错误模式** | 只自动更新项目级候选，团队级必须人工审核 |
| **中断预算过严导致误判推进** | 安全/合规/业务方向问题保留破例通道，并有审计记录 |
| **多仓路由误判导致漏看关键代码** | repo-router 只负责缩圈，不替代计划议会；高风险链路必须做 dependency map + 证据复核 |
| **跨仓索引失真或过期** | 在高风险任务前强制刷新目录树、关键配置、入口文件与依赖图 |

## 十七、度量面（新增）

为了让上述三项修订真正可验证，必须有量化指标：

| 指标 | 含义 | 目标 |
|------|------|------|
| `interrupt_count` | 实际强制中断次数 | 不超过预算 |
| `packet_compression_ratio` | 原始 must_confirm 数 / 最终 Packet 数 | 越高越好 |
| `avoidable_interrupts` | 本可假设推进却发起中断的次数 | 趋近 0 |
| `task_smoke_pass_rate` | task 级 smoke 通过率 | 稳定上升 |
| `stage_smoke_pass_rate` | 阶段级 smoke 通过率 | 稳定上升 |
| `replan_rate` | 因漂移/失败回流 PLAN 的比例 | 可控 |
| `candidate_skill_use_count` | 候选 Skill 被命中的次数 | 可观察 |
| `candidate_skill_success_rate` | 候选 Skill 应用成功率 | ≥ 0.8 更适合晋升 |
| `candidate_skill_rollback_rate` | 候选 Skill 导致回滚比例 | 趋近 0 |
| `repo_route_precision` | repo-router 命中仓库的准确率 | 稳定上升 |
| `cross_repo_deep_read_count` | 单 epic 深读仓库数量 | 尽量收敛 |
| `evidence_coverage_by_repo` | 每个命中仓是否都有证据定位 | 接近 100% |

这些指标统一写入 `.harness/features/{epic-id}/eval/metrics.json`。

## 十九、完整性补充（V4.2-final 补遗）

> 以下内容为对 V4.2 全文的系统性完整性审查后发现的缺口。对照前序文档链（`DEVELOPMENT-PLAN.md`、`final-plan-v2.md`、`final-plan-v2-supplements.md`、`ITERATION-PLAN.md`、`V4-final-plan.md`）逐项核对。按优先级分为三档。

### 20.1 关键缺口（不补会影响落地）

#### 20.1.1 DISCOVER 阶段缺失

V4.2 状态机为 `IDEA → CLARIFY → ... → DONE`，但前序文档（`final-plan-v2.md` 3.2 节、`DEVELOPMENT-PLAN.md` 2.4 节）均保留了 `DISCOVER → CLARIFY` 的可选前置阶段。

**场景**：用户扔进来的不是一个 feature 需求，而是一个过大或过模糊的目标（如"把整个系统从单体迁移到微服务"），此时需要先拆成多个可独立推进的 epic/feature 单元。

**建议补充**：

| 项目 | 内容 |
|------|------|
| **阶段** | DISCOVER（可选前置） |
| **工作流** | 1. 大目标分解 → 2. feature 列表 + 依赖关系 + 优先级排序 → 3. Decision Bundle（选择首批启动哪些 feature） |
| **产物** | `discovery-report.md`、`feature-list.json`（含依赖和优先级） |
| **出口** | 至少有一个 feature 具备进入 CLARIFY 的条件 |
| **门禁** | 无正式议会；Decision Bundle 即可 |
| **仓库介入** | 遵守 3.8 节分阶段策略，DISCOVER 阶段仅做 repo-router 级别快扫，不深读 |

修订后的状态机：

```
[DISCOVER →] IDEA → CLARIFY → SPEC → PLAN → EXECUTE → VERIFY → FIX → DONE
                                                    ↑        │
                                                    └────────┘
```

#### 20.1.2 人机交互协议缺失

V4.2 定义了 Decision Packet 的数据结构，但未说明**人怎样回答、在哪里回答、超时怎么处理**。`DEVELOPMENT-PLAN.md` 十一节有完整定义。

**建议补充**：

| 场景 | 交互方式 |
|------|---------|
| Decision Packet | CLI 对话窗口展示结构化 Packet，用户以 JSON 或选择式回答（如 `D-PLAN-03: FastAPI`） |
| 议会裁决 HOLD/BLOCK | 展示裁决报告摘要 + 关键阻断项，等待用户确认或补充信息 |
| FIX 轮次上限 | 展示问题汇总 + 三轮修复历史，升级为人工干预 |
| 回退确认 | 展示回退原因、影响范围、产物保留策略，等待用户确认 |

异步模式：
- 支持将 Decision Packet 写入 `.harness/features/{epic-id}/decision-packets/pending.json`
- 人工修改后通过 `/harness:advance` 命令继续
- 超时策略：设置最大等待时间，超时后标记为 `WAITING_HUMAN`，不自动推进

#### 20.1.3 回滚技术细节缺失

V4.2 反复强调"可回退"，但缺少回退的**具体操作规格**。`DEVELOPMENT-PLAN.md` 6.3 节有回滚策略表。

**建议补充**：

| 项目 | 策略 |
|------|------|
| 上游 artifacts 保留/清除 | **保留**（便于对比与审计） |
| 回退目标 | 回到上一阶段入口重新执行 |
| git 策略 | 从最后一次成功提交新建 fix 分支 |
| 状态追踪 | `state.json` 的 `rollbackHistory` 记录每次回退事件 |
| 次数限制 | 单阶段最多 3 次回退，超出升级人工干预 |

回退时 `state.json` 追加：

```json
{
  "rollbackHistory": [
    {
      "from": "EXECUTE",
      "to": "PLAN",
      "reason": "依赖顺序不成立，task fn-3.4 的前置 fn-3.2 被发现规格缺失",
      "timestamp": "2026-03-27T14:00:00Z",
      "preservedArtifacts": true,
      "gitRef": "abc123"
    }
  ]
}
```

#### 20.1.4 Hooks 技术契约缺失

V4.2 列出了 hooks 文件名，但没有给出 `hooks.json` 的完整配置和各 hook 的 stdin/stdout JSON 契约。`DEVELOPMENT-PLAN.md` 五节有详细定义。

**建议补充**：

`hooks/hooks.json` 推荐配置：

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/stage-init.sh"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/stage-gate-check.sh"
          }
        ]
      }
    ],
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/stage-reminder.sh"
          }
        ]
      }
    ]
  }
}
```

各 hook 的 stdout 契约：

| Hook | 输出字段 | 说明 |
|------|---------|------|
| `stage-init.sh` | `{ "additionalContext": "..." }` | 向 LLM 注入当前阶段上下文、待办、门禁要求 |
| `stage-gate-check.sh` | `{ "decision": "allow" }` 或 `{ "decision": "block", "systemMessage": "..." }` | 门禁通过或阻断 |
| `stage-reminder.sh` | `{ "additionalContext": "..." }` | 注入当前阶段简要提醒 |
| `runtime-preflight.sh` | `{ "decision": "allow" }` 或 `{ "decision": "block", "systemMessage": "..." }` | Task 开跑前检查 |
| `runtime-postcheck.sh` | `{ "additionalContext": "..." }` | Task 完成后 smoke/evidence 结果 |
| `interrupt-budget-check.sh` | `{ "decision": "allow" }` 或 `{ "decision": "block", "systemMessage": "..." }` | 中断预算是否允许发起中断 |

#### 20.1.5 状态机转换规则不完整

V4.2 画了状态机图，但没有显式列出所有**合法状态转换**和**触发条件**。

**建议补充**：

| 当前阶段 | 可转入 | 触发条件 |
|---------|--------|---------|
| DISCOVER | CLARIFY | feature-list.json 产出 + Decision Bundle 已处理 |
| IDEA | CLARIFY | `/harness:start` 或 `/harness:clarify` 调用 |
| CLARIFY | SPEC | Decision Bundle 已处理 + 中断预算已写入 |
| SPEC | PLAN | 轻议会 verdict = GO + Decision Bundle 已处理 |
| SPEC | CLARIFY | 轻议会 verdict = HOLD（回退） |
| PLAN | EXECUTE | 计划议会 verdict ≠ BLOCK + PLAN 控制包完整 |
| PLAN | SPEC | 计划议会 verdict = BLOCK 且原因为规格问题（回退） |
| EXECUTE | VERIFY | 全部 task 完成 + runtime receipts 齐全 |
| EXECUTE | PLAN | 漂移检测 / 依赖失败 / 连续失败 3 次（回退） |
| VERIFY | DONE | 技术 review SHIP + 验收议会 PASS + stage smoke 通过 |
| VERIFY | FIX | 验收议会 FAIL 或 stage smoke 失败 |
| FIX | VERIFY | 修复完成后重跑验收 |
| FIX | PLAN | 修复 3 轮仍失败 + 归因为计划问题（回退） |
| DONE | — | 发布议会 RELEASE_READY，流程终止 |

禁止的转换：
- 禁止跳过阶段（如 CLARIFY 直接到 PLAN）
- 禁止反向跳跃超过一个阶段（如 VERIFY 不得直接回 CLARIFY）
- 回退必须经过门禁确认，不得静默发生

### 20.2 重要缺口（不补会增加风险或降低效率）

#### 20.2.1 Token 成本预算缺失

V4.2 提到了动态强度控制，但没有给出**具体的 token 预算参考值**。`DEVELOPMENT-PLAN.md` 十二节和 `ITERATION-PLAN.md` 七节有详细数字。

**建议补充**：

| 环节 | 单次预估 Token | 频次 | 说明 |
|------|--------------|------|------|
| CLARIFY（brainstorming + interview） | ~15K | 1 次/epic | 含中断压缩 |
| SPEC（epic spec 生成 + interview 精炼） | ~30K | 1 次/epic | |
| SPEC 轻议会 | ≤ 50K | 1 次/epic | 3~5 reviewer |
| PLAN（scouts + task 图谱） | ~80K | 1 次/epic | scouts 并行调研 |
| PLAN 计划议会 | ≤ 150K | 1 次/epic | 5~7 reviewer，**最重的议会** |
| EXECUTE（单 task） | ~50K/task | N tasks/epic | worker + re-anchor + smoke |
| VERIFY 验收议会 | ≤ 200K | 1~3 次/epic | 5~7 reviewer，**成本最高** |
| DONE（发布议会 + 沉淀） | ~50K | 1 次/epic | 含候选 Skill 挖掘 |
| **单个 epic 总计** | **~600K~1.2M** | — | 取决于 task 数量和 FIX 循环次数 |

低风险场景通过动态强度控制可将总成本降至 ~300K。

单次 hook 执行时间硬限：≤ 5 秒（hook 只做文件检查和 JSON 读写，不调 LLM）。

#### 20.2.2 多 Epic 并行管理缺失

V4.2 的目录结构 `.harness/features/{epic-id}/` 暗示了多 epic 支持，但缺少显式的并行管理规则。`final-plan-v2-supplements.md` 2.3 节已指出此缺口。

**建议补充**：

并行管理规则：
- 每个 epic 拥有独立的状态机实例和 artifact 目录
- `stage-init.sh` 在 SessionStart 时检测所有进行中的 epic，优先恢复最近活跃的 epic
- 同一时间**只有一个 epic 可处于 EXECUTE 阶段**（避免代码冲突）
- 多个 epic 可同时处于 CLARIFY / SPEC / PLAN 阶段
- 跨 epic 依赖通过 `feature-list.json` 的 `dependencies` 字段管理
- 如果 epic-A 依赖 epic-B 的产出，epic-A 在 EXECUTE 前必须等 epic-B 至少进入 VERIFY

```json
{
  "activeEpics": ["fn-3", "fn-4", "fn-5"],
  "epicInExecute": "fn-3",
  "epicDependencies": {
    "fn-5": ["fn-3"]
  }
}
```

#### 20.2.3 MCP 配置策略缺失

`DEVELOPMENT-PLAN.md` 九节有 MCP 三层优先级策略。V4.2 的一体化插件架构下仍然需要 MCP 支持外部工具集成。

**建议补充**：

| 优先级 | 配置位置 | 用途 |
|--------|---------|------|
| 1（最高） | stage-harness 的 `plugin.json` 中 `mcpServers` | harness 专属 MCP 工具（如代码搜索、符号导航） |
| 2 | 用户级 `~/.claude.json` | 全局凭据与通用工具 |
| 3 | 项目级 `.mcp.json` | 项目特定的 MCP 集成 |

推荐用途：
- PLAN 阶段 scouts：通过 MCP 拉取外部文档、技术调研
- EXECUTE 阶段 worker：通过 MCP 执行构建、测试、部署
- DONE 阶段：通过 MCP 创建 PR、发布包、收集交付信息
- 多仓场景：repo-router 可通过 MCP 调用代码搜索引擎（如 Sourcegraph）加速跨仓符号导航

#### 20.2.4 Assumable 决策的追踪验证机制缺失

V4.2 定义了 `assumable` 分类和 `override_needed_by` 字段，但没有说明**系统如何在到达 needed_by 阶段时自动验证假设是否仍成立**。

**建议补充**：

Assumable 决策生命周期：

```
产生 → 记录假设依据 → 自动提醒（到达 needed_by 阶段时） → 人工确认或推翻 → 关闭
```

机制：
- 每个 assumable 决策的 `override_needed_by` 字段指定最晚确认阶段
- 对应阶段的 `stage-init.sh` 在加载时检查是否有到期的 assumable 决策
- 如有，注入 `additionalContext` 提醒 AI 和用户："以下假设即将到期，请确认或推翻"
- 如果用户未在到期阶段回应，该决策自动降级为 `must_confirm`（保守处理）
- 所有 assumable 决策的确认/推翻记录写入 `decision-bundles/` 的审计日志

#### 20.2.5 议会 Reviewer 冲突处理细则缺失

V4.2 定义了多数决、一票否决、加权多数三种策略，但未覆盖以下边缘场景。

**建议补充**：

| 场景 | 处理 |
|------|------|
| Reviewer 输出格式不标准（非结构化） | 降级为定性判断：含"PASS"/"通过"关键词视为通过，含"FAIL"/"不通过"/"阻断"视为不通过，其余标记为 `UNCLEAR` |
| 出现平票 | 安全相关议题：判为 FAIL（保守）；非安全议题：判为 PASS_WITH_CONDITIONS |
| Reviewer 超时未响应 | 等待 30 秒后跳过该 reviewer，以已收到的 reviewer 结果裁决；如有效 reviewer 数不足最低标准（轻议会 ≥ 2、其他 ≥ 3），整体标记为 `RETRY` |
| 多个 reviewer 结论自相矛盾 | 将矛盾点作为 `conditions` 写入裁决 JSON，verdict 降级为 `*_WITH_CONDITIONS`，由人工仲裁矛盾点 |

#### 20.2.6 Git 分支模型缺失

`DEVELOPMENT-PLAN.md` 十七节有完整的分支策略。V4.2 提到原子提交但没有系统性的分支模型。

**建议补充**：

| 维度 | 策略 |
|------|------|
| Epic 分支 | 每个 epic 一个分支（如 `feature/{epic-id}`），从主干创建 |
| Task 提交 | 每个 task 一个原子提交，commit message 格式：`[{epic-id}.{task-id}] {描述}` |
| 多仓场景 | 每个涉及的 repo 各自创建 epic 分支，跨仓 task 标注涉及的 repo + branch |
| VERIFY 合并 | VERIFY 通过后创建 PR 合并到主干 |
| FIX 提交 | 在同一 epic 分支上追加提交，commit message 标注 `[fix-{round}]` |
| 回退策略 | 从最后一次成功提交新建 fix 分支，原分支保留便于审计 |
| 自治模式 | 同一 epic 分支内操作，guard hooks 在安全检查点创建 tag |

### 20.3 建议缺口（锦上添花）

#### 20.3.1 Harness 自身测试策略

`DEVELOPMENT-PLAN.md` 十五节定义了 harness 自身的测试方法。stage-harness 作为编排层也需要自测。

**建议补充**：

| 测试类型 | 测试内容 | 方法 |
|---------|---------|------|
| **状态机单元测试** | 阶段流转、回退、循环终止、非法转换拒绝 | 模拟各种状态转换场景 |
| **门禁脚本集成测试** | 产物齐全/缺失、审查通过/失败、中断预算耗尽 | 准备各种通过/失败场景的 mock artifacts |
| **议会裁决测试** | 多数决/一票否决/加权多数/平票/超时 | Mock reviewer 输出 |
| **Hook 契约测试** | stdin/stdout JSON 格式是否正确 | 最小 PoC + 自动化回归 |
| **端到端冒烟测试** | 用最小 feature 跑通全流程 | "Hello World" 级别 feature 全流程 |
| **中断预算测试** | 预算计算、Packet 压缩、超限处理 | 模拟各风险等级的中断场景 |
| **多仓路由测试** | repo-router 命中准确率 | 用已知答案的多仓项目做回归 |
| **成本测试** | 各阶段 token 消耗是否在预算内 | 记录并对比多次运行数据 |

#### 20.3.2 安全层细化

V4.2 有安全 reviewer，但缺少对 **stage-harness 自身**的安全防护细节。`final-plan-v2-supplements.md` 3.5 节有具体建议。

**建议补充**：

| 威胁 | 防护 |
|------|------|
| Prompt Injection | MCP 输入输出做注入扫描；hooks 脚本不执行未转义的外部输入 |
| MCP 凭据外泄 | 最小权限原则；Secret 只在需要的阶段注入；不在 artifacts 中记录凭据 |
| 自治模式失控 | guard hooks 的 token 预算硬上限；连续操作次数上限；文件系统写入白名单 |
| 候选 Skill 注入恶意模式 | shadow validate 期间任何安全降级立即废弃；候选 Skill 不得包含 shell 命令或 MCP 调用 |

安全门禁在 DONE 阶段的发布议会中应显式包含：
- 依赖扫描（已知漏洞检查）
- Secret 检测（代码中是否泄漏凭据）
- OWASP 基线检查（如涉及 Web 应用）

#### 20.3.3 PLAN → EXECUTE 强门禁条件显式化

V4.2 在 3.1 节说"PLAN 是后半程控制中心"，但没有像 `final-plan-v2.md` 6.7 节那样显式列出 PLAN → EXECUTE 的**全部强门禁条件**。

**建议补充**：

进入 EXECUTE 前必须**同时满足**：

1. 计划议会 verdict ≠ BLOCK
2. task 图谱已成型，每个 task 有清晰范围、依赖、done 条件
3. 关键 task 的 TDD 策略不是泛泛"记得写测试"，而是有具体测试入口和验证靶子
4. 关键未决问题已清空或显式升级到 Decision Packet
5. 多仓任务已标注归属 repo 和接口边界
6. 中断预算已写入 `interrupt-budget.json`
7. `state.json` 记录本次 plan 放行结果与 conditions 项
8. 如 verdict = `READY_WITH_CONDITIONS`，conditions 项已写入跟踪列表

EXECUTE 的前提不是"计划写好了"，而是"**计划已具备执行控制力**"。

#### 20.3.4 降级策略（Graceful Degradation）

V4.2 是一体化插件，但内部各引擎（议会、runtime harness、多仓分析、skill evolution 等）可能在运行时出现异常。需要定义当某个引擎不可用时的降级路径。`ITERATION-PLAN.md` 八节有类似思路。

**建议补充**：

| 引擎异常 | 降级策略 |
|---------|---------|
| 议会 reviewer subagent 全部超时 | 降级为 checklist 模式（单 LLM 调用多视角审查），verdict 带 `degraded` 标记 |
| repo-router 无法识别仓库角色 | 降级为手动指定命中仓库；或退回到单仓模式 |
| runtime harness smoke 脚本不存在 | 跳过 smoke，在 receipt 中标记 `smoke: skipped`；不阻断但在 VERIFY 时降低置信度 |
| skill evolution 挖掘无结果 | 正常——不是每个 epic 都能产出候选 Skill |
| flowctl CLI 不可用 | 降级为直接读写 `.harness/` 目录下的 JSON 文件 |
| 跨模型 review 的外部模型不可用 | 降级为同模型 self-review；在 review report 中标记 `cross-model: false` |

核心原则：**任何单个引擎的故障不应导致整个流程不可用**。降级后继续推进，在度量面记录降级事件，VERIFY 阶段综合考量降级影响。

#### 20.3.5 上下文窗口预算量化

V4.2 的 8.4 节有"加载什么 / 不加载什么"的定性规则，但缺少量化预算。

**建议补充**：

| 阶段 | 预估上下文占用 | 硬上限 | 超限处理 |
|------|--------------|--------|---------|
| CLARIFY | ~10K tokens | 30K | 压缩历史对话，只保留最新一轮 |
| SPEC | ~20K tokens | 50K | 只加载 clarification-notes 摘要，不加载原始对话 |
| PLAN | ~40K tokens（含 epic spec + scouts 汇总） | 80K | scouts 报告做摘要压缩后再注入 |
| EXECUTE | ~25K tokens/task（含 task spec + epic 摘要 + 上轮 receipt） | 50K | 只加载当前 task 相关文件，不加载其他 task |
| VERIFY | ~60K tokens（含 epic spec + diff + evidence） | 100K | diff 超限时只加载变更文件列表 + 关键片段 |
| DONE | ~15K tokens | 30K | 只加载 verification summary 和模板 |

re-anchor 机制保证每个阶段入口都是"干净"的上下文加载，不残留前序阶段的噪声。

## 二十、结论

V4.2 的最终形态仍然是**一个自包含的 Claude Code 插件**——用户安装 stage-harness 后，丢进一段模糊需求，插件自动完成从需求澄清到代码交付的全流程。

在 V4.1 的 3 项定向补强基础上，V4.2 正式补入以下关键机制：

1. **多仓分析机制**——多代码仓库场景下，不做全仓硬扫，而是采用统一工作区、跨仓索引、repo-router、子代理下钻与证据式汇总。
2. **分阶段澄清机制**——需求澄清默认先脱离仓库做业务澄清，再结合仓库做落地澄清，最后只对高风险区域定点深挖。
3. **自动学习闭环升级为自动进化 Skill**——不只记经验，而是从运行记录中识别稳定模式，生成候选 Skill，并通过 shadow 验证与受控晋升迭代优化。
4. **运行时 harness 补强**——不只在阶段出口检查，而是在任务执行过程中持续做 preflight、smoke、triage、receipt、回流控制。
5. **最少打断变成硬约束**——通过中断预算、Decision Packet 批处理、预算超限压缩和度量面，让“少打断人”变成可执行、可统计、可审计的机制。

V4.2 补遗（第十九节）进一步覆盖了以下工程落地缺口：

6. **DISCOVER 前置阶段**——支持过大目标的拆解。
7. **人机交互协议**——Decision Packet 的回答方式、超时策略、异步模式。
8. **回滚技术规格**——产物保留、git 策略、次数限制、审计追踪。
9. **Hooks 技术契约**——完整的 hooks.json 配置和 stdin/stdout JSON 格式。
10. **状态机转换矩阵**——所有合法转换、触发条件、禁止转换。
11. **Token 成本预算**——各环节预估值和硬上限。
12. **多 Epic 并行管理**——并行规则、EXECUTE 互斥、跨 epic 依赖。
13. **MCP 配置策略**——三层优先级和推荐用途。
14. **Assumable 决策追踪**——到期提醒和保守降级机制。
15. **议会冲突处理**——平票、超时、格式异常、矛盾处理。
16. **Git 分支模型**——分支策略、提交规范、多仓协调。
17. **PLAN→EXECUTE 强门禁**——8 项显式准入条件。
18. **降级策略**——各引擎异常时的 graceful degradation。
19. **上下文窗口量化预算**——各阶段 token 占用预估和超限处理。

至此，V4.2（含补遗）覆盖了从战略设计到工程落地的完整规格。其余未修补项维持 V4 原判断与保留结论不变。
