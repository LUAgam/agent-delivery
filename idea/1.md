## 评估标准与边界

你要的不是“能写代码的 agent”，而是**阶段化 AI harness**：  
**模糊需求 → 澄清 → 方案/规格 → 实施计划 → 开发 → 测试 → 修复 → 交付**，并且关键点在于：

- **阶段产物可审查**（spec/design/plan/test evidence/review report/release notes）；  
- **强把控**：通过 reviewer + gate + hooks 硬阻断来防止“口头完成”；Claude Code 的 hooks 体系明确支持在 `TeammateIdle` / `TaskCompleted` 用 exit code 2 阻止闲置/阻止完成，还支持返回 JSON `{"continue": false, "stopReason": "..."}`。citeturn19view1turn19view2  
- **最小化人工交互**：只在阶段出口一次性返回“必须拍板才能前进”的决策（Decision Bundle），避免流程被频繁打断；  
- **可持续进化**：持续学习要“可控”，更推荐“候选学习→人工晋升”而不是自动改规则；ECC 的 continuous-learning 提供了明确的“从会话提炼可复用技能”的设计方向。citeturn16view0turn16view1  
- **少量二开**：优先通过 Claude Code 插件（skills/agents/hooks/MCP）实现，而不是一开始就上 Agent SDK 平台化。官方文档明确 plugins 可打包 skills、agents、hooks、MCP servers。citeturn3search17turn4search12  

边界提醒（非常重要）：  
- **Claude Code 的“插件 subagent”出于安全原因不支持 hooks / mcpServers / permissionMode frontmatter（会被静默忽略）**，如果你需要这些字段，需把 agent 放到项目 `.claude/agents/` 或用户 `~/.claude/agents/`。这会影响你“仅靠插件”做细粒度控制的方式：你要么在全局 hooks 中用“读 state 文件 + 判阶段”来控工具，要么在安装脚本中把 agent 写入 `.claude/agents/`。citeturn13view6

## 新发现的更合适平替候选

下面按“更像你的目标（阶段化+把控+少二开）”来分类。

### 框架级平替：SuperClaude Framework

**定位与亮点**  
SuperClaude Framework 明确自我定位为“增强 Claude Code 的配置框架”，包含大量命令/agents/modes/MCP，并提供安装指南与命令参考文档。citeturn1view1turn0search4turn1view4  
其插件仓库给出了统计：**29 commands、23 agents、7 modes、10 MCP tools**，并强调“通过行为指令注入与组件编排，把 Claude Code 变成结构化开发平台”。citeturn1view2  

**对你诉求的适配度**  
- 更接近“现成工作台”：命令分类、典型工作流、会话启动恢复说明、MCP 组合策略都已经文档化（例如 commands reference 里提到的工作流与 MCP server 列表）。citeturn1view4  
- 但它的安装/运行方式更像你在 `~/.claude/` 里落地一套体系（而不是轻量插件），并且明确提醒“会修改 Claude Code MCP 配置，安装前必须备份”——这对你“风险低、可控集成”的目标是一个现实成本。citeturn1view2  

**推荐用法**  
- 不建议作为“一键平替”直接替换你当前组合；  
- 更建议作为**参考架构与模板库**：借用其命令分类、SessionStart 协议、MCP 组合与“职责分工”的写法，把这些理念固化到你的集成插件方案 A 中。citeturn1view4  

### 端到端插件三部曲平替：Deep Trilogy

这套是这次调研里最像你“从模糊到交付”的一条龙平替。

- **/deep-plan**：明确说明“把模糊需求通过研究、访谈与多模型 review 变成 production-ready implementation plan”；并说明“大项目先用 /deep-project 拆分，再逐组件 deep-plan”。citeturn4search0turn4search4  
- **/deep-implement**：明确定位为“消化 deep-plan 产出的 sections”，以 **TDD + code review + atomic commits** 逐段实现；强调可恢复、每段实现/评审/提交再进入下一段。citeturn4search1turn4search4  

这套三部曲对你两个关键痛点非常对位：  
1) **计划阶段的“外部 review + TDD 结构化计划”**（比一般 /plan 更强的“多模型审阅+分段交付”心智模型）；citeturn4search0turn4search4  
2) **执行阶段的“原子提交与可恢复流水线”**（减少一次性大提交、降低回滚成本、减少中途上下文断裂）。citeturn4search1turn4search4  

**结论**：Deep Trilogy 是目前最强的“局部平替”候选：  
- 你可以把它作为 **PLAN→EXECUTE** 段的替代实现；  
- 或者吸收它的“多模型 plan review + 分段实现 + 原子提交”的范式，落地到你自己的插件流程中。citeturn4search0turn4search1turn4search4  

### 持续学习/复盘平替补充：aiocean/claude-plugins（尤其 aio-reflect）

aiocean/claude-plugins 是一个“plugin 集合”，列出多类插件：codebase 分析、planning、debug、code review、worktree、以及最关键的 **aio-reflect**（会话复盘）与 **aio-claude-manager**（按项目上下文启用/禁用 skills）。citeturn5view0

这与 ECC 的 continuous-learning（从会话提炼可复用 skills/instincts）是同方向，但实现路径不同：ECC 更像“harness 性能系统”；aio-reflect 更像“轻量复盘与生成器”。citeturn16view0turn16view1  

### “复利工程”平替/补强：EveryInc/compound-engineering-plugin

EveryInc 的 Compound 工程插件对你“持续学习 + 少打断 + 长期演进”特别有价值，原因是它把工作流明确写成闭环：

### 最高收益的替换建议

**把 PLAN→EXECUTE 这一段，用 Deep Trilogy 替换/增强**

## 风险与注意事项

### 配置侵入性与回滚风险

SuperClaude 插件明确警告它会修改 Claude Code MCP 配置，并要求先备份以便回滚。若你已经在做“自建集成插件”，更推荐把 SuperClaude 当“参考库”，而非直接全量安装覆盖。citeturn1view2  

### agent teams 与 hooks 的工程边界

- agent teams 是实验特性且成本更高，适合只放在少数“关口审议”。  
- `TeammateIdle` / `TaskCompleted` 不支持 matcher，会每次触发；这非常适合做“硬 gate”，但你必须设计好**避免误伤**（例如读取 `.harness/state.json` 只在某些 stage 执行阻断）。citeturn19view1turn19view2  

### 插件 subagent 的安全限制

再次强调：插件 subagent 的 hooks/mcpServers/permissionMode frontmatter 会被忽略。你最好在方案 A 里明确两种模式：  
- **纯插件模式**：所有控制放在全局 hooks + permissions + state gating；  
- **增强模式**：提供 `setup` 命令把 agents 下发到项目 `.claude/agents/`，以启用更细粒度字段。citeturn13view6  

### 安全补强建议

你在用 MCP 和自动执行时，建议把 “prompt injection / 机密外泄” 当一等风险：  
- Claude Code 官方 MCP 文档有专门的安全提醒（第三方 server 风险、最小权限）。citeturn7view1  
- 社区也出现专门的“注入扫描 hooks/工具”（例如 awesome list 中提到的 injection scanner）。citeturn4search17turn1view0
